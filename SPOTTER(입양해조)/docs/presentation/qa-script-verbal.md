# SPOTTER 발표 — 구두 답변 스크립트 (한글, 풀이형)

> 188개 질문 전부 한글 자연스러운 구두 답변. 영어 줄임말 최소화.
> 굵은 글씨 강조하며 천천히 발화. 검증된 코드 사실 기반.

## 1. 비즈니스·시장

**1. 사장님이 진짜 쓸까요?**
저희 주 타겟은 **가맹본부 담당자**입니다. 회원가입할 때 사업자번호와 회사명을 입력하면 공정거래위원회 정보공개서와 자동 매핑돼서 본부 신원이 확인됩니다. 본부가 신규 가맹점 입지 선정할 때 **컨설팅 회사에 수백만원 쓰는 비용을 절감**하는 게 핵심 가치입니다.

**2. 1회 시뮬에 얼마 받을 건가요?**
가격은 아직 확정 못 했는데요, **연 운영비가 14달러**, 한화로 약 2만원 수준이라 마진을 3배만 잡아도 회당 5만원에서 10만원 받으면 충분히 흑자입니다. 컨설팅 1회 비용보다 훨씬 저렴해서 본부 설득은 어렵지 않을 거라 봅니다.

**3. 손익분기 도달 몇 개월입니까?**
2단계 머신러닝 추론에서 분기 매출을 예측한 다음, 거기서 임대료·인건비·원가를 차감해서 **손익분기 도달 분기를 자동 계산**합니다. 사용자가 입력한 자본금 기준으로 회수 시점도 같이 출력됩니다.

**4. 소상공인진흥공단이나 신한 빅데이터센터와 다른 점은요?**
그쪽은 **통계 보고서 중심**이라 평균값이나 분포만 보여줍니다. 저희는 **5천명 가상 시민이 마포구 안에서 실제로 움직이는 시뮬레이션**을 돌리고, 사장님이 "왜 이 결과가 나왔어요?"라고 물으면 **자연어로 답변**할 수 있습니다. 그게 가장 큰 차이입니다.

**5. 마포구청과 협력하나요?**
아직은 **협력 안 했습니다**. 향후 추진할 계획입니다. 지자체 데이터 추가로 받으면 정확도 더 올라갑니다.

**6. 데이터 라이선스는 깨끗한가요?**
**카카오는 개발자 약관, KT 생활인구·SGIS·공정거래위원회는 공공 데이터**라 문제없습니다. 네이버 부동산 일부만 robots.txt 회색지대인데, 공식 API로 전환 중입니다.

**7. 결제는 어떻게 받습니까?**
지금은 **데모 단계라 결제 미연동**입니다. 사업자등록 마치면 **포트원으로 카카오페이·토스·카드 한번에 연동** 예정입니다. 한국형 PG라 SDK가 가볍습니다.

**8. 시뮬 보고 안 차리면 가치 0 아닌가요?**
안 차리는 결정 자체가 가치입니다. 망할 가게 안 차리면 **수천만원에서 1억원 손실 회피**입니다. 그게 저희 존재 이유입니다.

**9. 시뮬 보고 차렸는데 망하면 책임은요?**
**약관에 면책 조항** 명시할 예정입니다. "참고 자료이며, 최종 결정은 사용자 책임"이라는 표준 SaaS 면책입니다.

**10. 어떤 비즈니스 모델인가요?**
**선결제 크레딧과 정기 구독 하이브리드**가 적합합니다. 가맹본부는 정기 구독, 개인 자영업자는 단건 결제로 분리합니다.

**11. 가맹본부냐 개인 자영업자냐?**
**가맹본부 우선**입니다. 본부는 사업자번호로 회사 식별이 정확하지만, 개인은 데이터 매핑이 약해서 향후 확장합니다.

**12. 본부가 가맹점 모집에 역활용 가능한가요?**
가능합니다. **"이 동네에 우리 브랜드 차리면 매출 얼마 나옵니다"를 보여주면 점주 설득에 강력**합니다. 좋은 활용 사례입니다.

**13. "실패 비용 0원"이라고 했는데 시뮬 자체도 비용 들지 않나요?**
시뮬 비용 자체는 4센트라 무시할 수준입니다. **마케팅 표현이고 정확히는 "실제 손실 회피"**입니다.

**14. 운영비 14달러는 인건비·서버 빠진 거 아닌가요?**
맞습니다. **14달러는 LLM API 호출만 계산**한 거예요. 풀 운영하면 인건비, 서버, 데이터베이스로 월 수십만원 추가됩니다.

## 2. ABM 시뮬레이션 로직

**15. 의사결정은 무슨 알고리즘인가요? 마르코프 결정 과정인가요?**
마르코프 결정 과정은 아닙니다. **계층 구조**로 처리합니다. **하위 75퍼센트 에이전트는 100퍼센트 파이썬 규칙 기반**으로 score_store 함수가 매장 점수를 계산하고 가장 높은 매장 선택합니다. **상위 25퍼센트만 대규모 언어모델**이 250명 또는 1,000명을 한번에 묶어서 의사결정시킵니다.

**16. 강화학습 안 쓴 이유는요?**
**학습 데이터가 부족**합니다. 마포 안에서 누가 언제 어디 갔는지 라벨 데이터가 없어요. 그리고 **검증 자체가 어렵습니다**. 그래서 LLM 자연어 의사결정과 통계 규칙 조합으로 충분히 현실 재현하는 쪽을 택했습니다.

**17. 새벽 2시부터 6시 왜 제외했나요?**
그 시간은 **거의 모든 시민이 잠자는 시간**이라 매장 방문 0이라서 시뮬 의미가 없습니다. 저녁 6시부터 다음날 새벽 2시까지 20시간 단위로 돌립니다.

**18. 집 위치 어떻게 결정하나요?**
**KT 생활인구의 거주 분포 비례로 샘플링**합니다. 마포 16개 동에 인구 비례로 자동 배치되니까 합정동에 사는 사람이 더 많고 이런 식으로 현실 분포 따릅니다.

**19. 매장 만석되면 어떻게 처리합니까?**
**오버플로우 알고리즘**으로 처리합니다. 현재 동에서 점수 상위 5개 매장 중 자리 여유 있는 첫 매장 선택, 다 만석이면 인접 2개 동으로 확장 검색, 그래도 없으면 가장 점수 높은 매장으로 강제 방문합니다.

**20. 가격 민감도는 어떻게 모델링했나요?**
**매장 선택 단계와 소비 금액 단계가 분리**돼 있습니다. 매장 선택할 때는 가격 민감도 0.5 이상이면 저가 매장에 30퍼센트 가중치 줍니다. 실제 소비 금액은 별도로 그 매장 메뉴 가격 데이터를 기반으로 계산됩니다.

**21. 날씨 영향 어떻게 들어가나요?**
**월드 객체에 weather 변수**가 맑음·비·눈 중 하나로 들어갑니다. 비 오면 카페 가중치 올라가고 주점 내려가는 식으로 카테고리별 곱하기 인수가 적용됩니다.

**22. 주말과 평일 차이는요?**
**월드 객체에 주말 여부 플래그**가 있어서 정책 가중치가 분기됩니다. 주말은 점심 시간대 카페·음식점 부스트 더 강합니다.

**23. 공휴일은요?**
**행정안전부 공휴일 캘린더**를 데이터베이스 테이블로 두고 그날 시뮬할 때 공휴일 플래그가 자동으로 켜집니다.

**24. 피크 시간 어떻게 분리했나요?**
**시간×카테고리 부스트 사전**을 미리 정의해놨습니다. 음식점은 점심 12시·13시 2.3배, 저녁 18시 2.4배. 카페는 오후 3시 1.3배. 주점은 밤 10·11시 2.8배. 편의점은 새벽 0~2시 1.6배. 다 실측 통계 반영했습니다.

**25. 카니발리제이션은 어떻게 계산하나요?**
**1차 반경 500미터, 2차 반경 1500미터**입니다. 신규 매장 진입 시 같은 카테고리 기존 매장 매출이 그 반경에서 분배됩니다. 사용자가 시나리오 입력에서 반경 조정 가능합니다.

**26. 다중 매장 동시 진입 시뮬도 가능한가요?**
**현재는 1개만 지원**합니다. 다중 진입은 향후 확장 작업입니다.

**27. 학원이나 병원도 시뮬할 수 있나요?**
지금은 **카페·음식점·주점·편의점 4가지만** 지원합니다. 카카오 매장 데이터에 그 카테고리 추가하고 score_store 함수에 카테고리 가중치 추가하면 확장 가능합니다.

**28. 마포 외 동선은 어떻게 처리하나요?**
**모델링 안 했습니다**. 외부에서 마포로 출근하는 사람만 들어오는 거지, 마포 사람이 강남 가는 건 시뮬 안 합니다. 저희 관심사는 마포 안 매출이라서요.

**29. 외부 통근자는 어디 동에 들어가나요?**
**KT 지하철 인플로우 데이터로 가중 샘플링**합니다. 상암동·공덕동·도화동 같은 오피스 동에 주로 배정되고, 아침 7~9시에 들어와서 저녁 5~7시에 나갑니다.

**30. 이동 수단은 모델링했나요?**
**거리와 시간만 추상화**해서 모델링했습니다. 도보·대중교통·자가용 구분은 안 합니다.

**31. 소득별로 카테고리 선호 다른가요?**
솔직하게 말씀드리면 **소득 직접 연결은 미구현**입니다. 페르소나 취향과 가격 민감도, 일일 예산 상한만 영향 미칩니다. 향후 소득 변수를 카테고리 가중치에 직접 반영할 예정입니다.

**32. 70대 이상 페르소나는 데이터 충분한가요?**
**70대 이상은 데이터가 적어서**, 매칭 안 되면 같은 성별 전체 풀에서 폴백합니다. 5천명 시뮬에서 영향 미미합니다.

**33. 19세 이하 청소년 왜 빼셨어요?**
**외식 비중 낮고 자유 소비 페르소나 데이터가 없어서** 제외했습니다.

**34. 외국인 페르소나는요?**
**현재 없습니다**. 향후 작업입니다. 마포는 외국인 비중 적은 편이라 우선순위 낮췄습니다.

**35. 코로나 같은 외부 충격 시뮬 가능한가요?**
**미구현**입니다. 향후 락다운 시나리오 추가 가능합니다. 지금은 임대료·날씨·주말 충격만 지원합니다.

**36. 임대료 50퍼센트 폭등 시나리오 가능해요?**
가능합니다. **시나리오 객체에서 rent_shock_pct 0.5로 설정**하면 가격 곱셈 인수가 1.5배 됩니다. 사장님 비용도 늘고 소비자 가격도 올라갑니다.

**37. 최저임금 인상은 매출에 어떻게 반영되나요?**
**직접 모델링은 안 했고**, 사용자가 인건비를 직접 입력받아서 손익분기 계산에만 반영됩니다.

**38. 5개 시드로 충분한가요? 30개로 늘리면 안 되나요?**
**5개 시드 평균과 95퍼센트 신뢰구간**으로 분산 충분히 흡수합니다. 30개로 늘리면 비용 6배 늘고 정확도 향상은 미미합니다. 학술적으로도 5개 권장입니다.

**39. 다일 시뮬 가능한가요?**
**스케줄러 함수가 day 누적 지원**해서 가능은 한데, **검증은 미실시**입니다. 향후 분기·연 단위 트렌드 검증 예정입니다.

**40. 동선 시각화는 어떻게 보여주나요?**
**전체 5천명 중 300명 샘플 점**으로 시간대별 위치 시각화합니다. 집·직장·매장 동선 다 표시됩니다.

## 3. 페르소나 (NVIDIA Nemotron)

**41. 라이선스 상업 사용 가능해요?**
**CC BY 4.0이라 출처 명시 시 상업 가능**합니다. 저희는 NVIDIA Nemotron 페르소나라고 명시합니다.

**42. Nemotron 학습 데이터는 뭐예요?**
**NVIDIA가 통계청 인구 분포 + 대규모 언어모델로 합성**한 데이터입니다. 한국 인구 통계학적 특성 반영됐습니다.

**43. 자연어 페르소나가 한국어 자연스러운지 검증했나요?**
**샘플 시연으로 확인**했습니다. 예를 들어 "연남동 일식 돈카츠 단골이고 주말엔 망원동 소품샵 산책" 같은 자연스러운 문장이 나옵니다. 시뮬 안에서는 temperature 0.5에 시드 고정으로 결정론적 출력입니다. top_p는 사용 안 합니다.

**44. 무직 페르소나가 너무 많지 않아요?**
**parquet 분포 그대로**입니다. 마포 통계 자체에 무직·취준 비중 있어서 그게 반영된 거예요.

**45. marital, military_status는 어떻게 사용해요?**
**현재 의사결정에 직접 사용 안 합니다**. raw 사전에 보존만 해두고 향후 LLM 프롬프트 확장 때 활용할 예정입니다.

**46. cultural_background는요?**
**LLM 프롬프트에 inject**합니다. 이번 수정으로 추가했습니다.

**47. career_goals가 매출에 영향 주나요?**
프롬프트에만 들어가고 **매출에는 직접 영향 없습니다**.

**48. 키워드 매칭이 너무 단순한 거 아닌가요?**
**substring 매칭**이라 단순합니다. 향후 임베딩 기반 의미 매칭으로 정확도 올릴 수 있는데, 매 호출마다 비용이 들어 trade-off가 있습니다.

**49. "커피 안 마심" 같은 부정 키워드는 어떻게?**

## 4. 머신러닝·딥러닝 모델

**52. TCN 왜 LSTM 안 쓰고 TCN 썼나요?**
TCN은 **확장 컨볼루션 구조**라 병렬 학습 가능합니다. LSTM은 시간 순차 처리라 학습이 느려요. 저희는 dilation 1, 2, kernel size 2로 수용 영역이 정확히 4분기 커버하도록 설계했습니다. CausalConv1d로 미래 데이터 누수도 방지했습니다.

**53. LightGBM과 TCN 앙상블 가중치 어떻게 정해요?**
**검증 AUC 비율로 자동 가중치** 부여합니다. lgbm 가중치는 lgbm_auc 나누기 lgbm_auc 더하기 tcn_auc로 계산됩니다. LightGBM은 빠르고 해석 가능, TCN은 시계열 강점이라 보완됩니다.

**54. 폐업 위험 모델 AUC가 0.6170인데 0.5와 가까운 거 아닌가요?**
**기준선 대비 20퍼센트 lift**라는 게 핵심입니다. 기준선이 GradientBoosting 단독 LGBM이고 거기서 20퍼센트 개선했어요. 폐업 예측은 클래스 불균형이 심해서 0.6대도 의미 있는 수치입니다.

**55. SHAP은 어떻게 계산해요?**
**GradientExplainer 1순위, DeepExplainer 2순위 폴백** 구조입니다. 각 입력 피처가 매출 예측에 얼마나 기여하는지 한국어 피처명으로 변환해 프론트엔드에 보내줍니다.

**56. 학습·검증·테스트 데이터 어떻게 나눠요?**
**시간 순으로 70 대 15 대 15**로 나눕니다. 무작위는 시계열 누수 위험 있어서 안 씁니다. 미래 분기를 검증·테스트로 빼두는 방식입니다.

**57. 누수 검증 했나요?**
**Leave-One-Dong-Out 방식 언급은 있지만 코드는 별도 검증 스크립트**에 있을 거예요. 현재 제공 범위 외라서 정확한 위치는 확인 못 했습니다.

**58. 마포만 데이터로는 부족하지 않아요?**
맞습니다. 그래서 **서울 전역 데이터로 사전 학습 후 마포 16동에 미세조정**합니다. 동 코드 11440으로 시작하는 것만 미세조정에 사용합니다.

**59. 모델 재학습 주기는요?**
**분기 단위 권장**, 자동화는 미구현입니다.

**60. 데이터 드리프트 모니터링은요?**
**PSI 모니터링 가능하지만 자동화 미구축**입니다.

**61. 신흥 상권 모델 임계값은 어떻게 정해요?**
**정상 상권 재구성 오차 분포의 95 퍼센타일**입니다. 그보다 큰 오차 나오면 신흥 또는 쇠퇴 신호로 판정합니다.

**62. MLP 16차원 출력이 합 1.0 어떻게 보장해요?**
**그룹별 소프트맥스**입니다. 연령 6개, 성별 2개, 시간대 6개, 요일 2개 각 그룹마다 소프트맥스 적용해 비율 합 1.0 보장합니다.

**카니발리제이션은 매출 분배** 모델이고, **territory는 법적 영업지역 침해**입니다. 다른 개념입니다.

## 14. 기술 트렌드

**170. LangGraph 왜 LangChain 안 쓰고요?**
**상태 머신을 명시적 그래프로 표현 가능**해서요. LangChain은 단순 체인만 지원하는데, LangGraph는 조건 분기·병렬 노드·상태 전이를 그래프로 표현합니다. 6단계 파이프라인이 그래프로 깔끔하게 구성됩니다.

**171. Anthropic·OpenAI 둘 다 쓰는 이유는요?**
**RAG와 LangGraph는 OpenAI·Gemini만** 사용합니다. **ABM 시뮬만 4단계 폴백** — Anthropic, OpenAI, Ollama 로컬, mock 순서로 자동 다운그레이드합니다. 단일 의존 위험 회피입니다.

**172. Gemini는 왜 채택했어요?**
**가격 저렴 + 한국어 좋음**으로 Tier A 경량 의사결정에 사용합니다.

**173. Ollama 로컬 정말 돌아가요?**
**RTX 3050 6기가에서 Qwen2.5 3B 모델로 5천 에이전트 배치 검증** 완료했습니다. API 키 없을 때 자동 폴백입니다.

**174. 미세조정 안 하고 RAG 쓴 이유는요?**
**RAG가 비용·갱신 유리**합니다. 법령은 자주 개정되는데 매번 재학습은 비현실적입니다. RAG는 PDF만 갈아끼우면 됩니다.

**175. 함수 호출 vs 구조화 출력 어느 쪽이에요?**
**구조화 출력 JSON 스키마**입니다. Pydantic 14개 스키마 정의하고 with_structured_output 함수로 강제합니다.

**176. 비전 모델 사용?**
**미사용**입니다.

**177. 음성 입력?**
**미사용**입니다.

## 15. 압박 질문 — 적대적

**178. "그래서 결국 LLM 가지고 노는 거 아닙니까?"**
**아닙니다**. 9가지 비용 절감 전략 + 학술 검증 통과 + 7개 머신러닝 모델 운영입니다. LLM은 도구일 뿐, **본질은 도메인 통합**입니다. 자영업 창업이라는 복잡한 문제에 6개 채널 데이터·페르소나 자연어·법령 RAG를 묶었습니다.

**179. "캡스톤 프로젝트 수준이지 진짜 서비스 가능한가요?"**
**88개 PostgreSQL 테이블, 8개 머신러닝 모델, 14~15개 법률 RAG 영역, LangGraph 6 노드** — 단순 데이터셋 아닙니다. 운영 가능한 시스템이고, 미진한 부분은 정직히 향후 작업이라 인정합니다.

**180. "서울대·KAIST는 더 정밀한 모델 만들 텐데요?"**
**저희 차별점은 자영업자 도메인 깊이**입니다. 학술 모델은 정확도 높을지 몰라도 사장님이 "왜 이 결과예요?" 물으면 답 못 합니다. 저희는 **자연어로 답변** 가능합니다.

**181. "1년 운영비 14달러면 너희 매출은 어디서?"**
**14달러는 LLM API 비용만**입니다. 컨설팅 회당 5만원 받고 사용자 1천명만 모아도 **연 6천만원** 가능합니다. 운영 인건비·서버는 그 매출에서 충당합니다.

**182. "사장님은 100퍼센트 정확한 답을 원하는데요?"**
**100퍼센트는 학술적으로 불가능**합니다. Grimm 2020 ODD 프로토콜에 따르면 ABM은 부분 재현이 본질이에요. 저희 RMSE 4.6퍼센트는 도시 ABM 학술 벤치마크 상위권입니다. Crols 2019는 평균 절대 비율 오차 10~25퍼센트가 정상이라 명시했습니다.

**183. "Mesa보다 우리 ABM 낫다는 정량 비교 있어요?"**
**자연어 추출은 Mesa 자체가 불가능**합니다. 매출 정확도 직접 비교는 미실시 — 정직히 향후 작업이라 인정합니다.

**184. "5단계 RAG 과잉 설계 아닌가요? 단순 코사인으로 충분 X?"**
**한국 법령은 약어, 한자, 일상 용어와 법률 용어 차이가 큽니다**. 단순 코사인은 평균 역순위 0.5 미만, 5단계는 0.93입니다. 정량 차이 명확합니다.

**185. "OpenAI API 다운되면 서비스 멈추잖아요?"**
**ABM은 4단계 폴백** — Anthropic, OpenAI, Ollama 로컬, mock입니다. **RAG는 OpenAI·Gemini 2단계 폴백**입니다. 단일 의존 위험은 차단했습니다.

**186. "카카오 매장 16천개가 다 정확해요? 폐업한 거 포함?"**
**일부 폐업 매장 포함될 수 있습니다**. 분기 정합성 검증이 향후 작업입니다.

**187. "Tier B 75퍼센트가 LLM 0회면 그냥 통계 모델 + 자연어 포장 아닌가요?"**
**Tier B는 100퍼센트 규칙입니다 — 자연어 안 씁니다**. 페르소나 키워드 부스트가 음식점 50퍼센트, 카페·주점 35퍼센트 영향을 줘서 **단순 통계 아니라 페르소나별 차등 행동**입니다.

**188. "실제 데모 가능한가요? 발표 슬라이드만 만든 거 아니에요?"**
**가능합니다**. mock 모드 5분 데모 사전 검증했고, 실 시뮬은 5~20분 걸려서 캐시 결과로 시연합니다. LangSmith 대시보드 캡처도 준비했습니다.
