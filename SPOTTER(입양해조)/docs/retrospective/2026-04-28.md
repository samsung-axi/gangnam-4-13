## 2. Keep (계속 유지)

### 분리 표시 (Sprint 4)
일반 셀과 외삽 셀을 별도 confidence base 로 측정하는 design 이 가장 정직하면서 합격 가능했다. 향후 imputation 작업의 표준 패턴으로 채택.

### Multi-Output ExtraTrees
48 컬럼 동시 학습 + 6 seed 앙상블 + raking post-processing 조합이 우리 데이터 규모에 최적. SOTA 라이브러리 (HyperImpute, LightGBM) 보다 우수.

### Sum constraint raking
5종 sum 제약 (weekday/요일/시간/성별/연령) 을 raking 으로 강제 → 합계 일관성 0.000016% 오차 보장. 다운스트림 어디서 써도 정합성.

### 코드 리뷰 두 단계 (spec + quality)
Spec compliance reviewer + Code quality reviewer 분리 dispatch 가 issue 발견에 효과적. Sprint 7 에서 HIGH 7 건 잡음.

### 정직 보고 = Done 정의
spec 의 "fail 시 정직 보고가 done" 정의가 본질적 한계 인정 + 진행 가능하게 함. production-ready 자체를 done 으로 잡았다면 무한 반복.

## 3. Problem (개선 필요)

### Code reviewer 의 false positive
Sprint 1 Task 3 reviewer 가 "dong_avg leak" 을 HIGH 로 지적 → Sprint 5 에서 4시간 학습 후 가설 기각. Reviewer 의 심각도 판정이 데이터 검증 없이 진행됨.

### 합격선 자체의 모순
- 4-1 sensitivity ≥ 8% — imputed 가 정확할수록 sensitivity 가 작음 (값이 기존과 비슷). 8% 변화는 imputation 부정확 의미.
- 1-4 confidence ≥ 0.75 (전체 평균) — 외삽 48셀 (35%) 이 0.4 cap 으로 평균 끌어내림 → 구조적 불가능.

### 디스크 공간 의존성
BEST_PARAMS (n=300, depth=35) 6 seed 학습이 48GB 필요. 사용자가 정리하지 않으면 sprint 진행 불가. 처음부터 디스크 요건 명시했어야.

### KOSIS API 한계
DT_1KC2023 단일 anchor (시도×분기) 의 동/업종 해상도 차이 2단계. 추가 anchor 시도 (DT_1K41017, DT_3KB9001) 도 모두 실패.

### MNAR-mimic CV 의 leak 이해 불충분
Sprint 1 Task 3 가 LOO 효과를 "delta 0.05%p" 로 측정해 leak 영향 미미라 결론. 그러나 Sprint 5 K-fold cross-fitting 적용 시 +7%p 악화 → 두 LOO 방식이 다른 효과. CV 설계의 ablation 이 깊이 부족.
