# 2026-05-01 회고 — closure_risk C layer (label) fix + production retrain

## 요약

폐업 위험도 모델의 label 정의를 **3중 OR (closure_rate spike / store_count 2분기 연속 감소 / sales YoY -25%) → 단일 quantile 조건 (`next_closure_rate > industry_p75_train`)** 으로 단순화. `industry_p75` 를 **train split rows 만으로 fit** 하여 leakage 차단. spec → plan → 5-task subagent + 1 inline 실행 → production retrain 완료. **결과: test AUC 0.5142 → 0.5983 (+0.084), val/test gap 0.026 → ~0**. label 정의의 정직성 향상이 의미 있는 신호 회복으로 이어졌다.

---

## 1. 배경

2026-04-30 sprint 에서 검증 layer (E) 를 random split → time-based holdout 으로 교체하고 정직한 metric 측정 가능해졌다. 그 결과 ensemble test AUC = 0.5142 (random) 로 모델이 **production-ready 가 아님** 이 확정됐다.

진단 결과 5 layer 중 **C (label) 의 비중이 가장 큼** 으로 식별:
- `data_prep.py:189` 의 3중 OR 가 무엇을 예측하는지 모호 — pos_ratio 0.27~0.32 의 출처 분해 불가
- `industry_avg = transform("mean")` (`data_prep.py:177`) 가 **train+val+test 통합 평균** → 약한 leakage
- row 단위 (dong × industry × quarter) macro 신호 — 본 sprint 에서는 그대로 유지 (별도 spec)

**가설**: label 단순화 + leakage 차단 시 (1) 모호함 제거로 신호 명확화, (2) train/val/test 일관성 향상으로 generalization 개선.

부수효과: val 표본이 더 커져 metric 추정이 더 안정적.

### 진단

- **Test AUC 0.5983** — random (0.5) 와 의미 있는 차이. 하지만 random split 인플레 (0.85) 와는 여전히 거리. **첫 production-ready 신호 회복** 으로 평가.
- **Val/Test gap 사라짐** — label 정의가 일관되어 train 에서 val/test 로 generalization 잘됨. 이전 0.026 gap 은 label 모호함 (3중 OR) + 약한 leakage 의 합산이었던 것으로 추정.
- **PR-AUC 약간 하락 (-0.016)** — pos_ratio 가 0.32 → 0.22 로 줄어 baseline 도 동시 하락. lift (PR-AUC / pos_ratio) 로 보면: 이전 0.331/0.323 = 1.025× → 이번 0.315/0.222 = 1.418×. **신호 강도는 더 강해짐**.
- **Brier 0.192** — 확률 보정이 좋아짐. label 단순화로 예측 확률이 더 명확한 의미 (industry p75 통과 여부) 를 가지게 된 영향.
- **결론**: production-ready 한 약한 신호 확보. AUC 0.60 수준은 의사결정 보조용 (e.g. 위험 후보 top-K 추천) 으로는 사용 가능, 단독 의사결정 (binary 차단) 에는 부족.

## 5. 배운 점

1. **Label 단순화의 정직한 효과** — 3중 OR 가 신호를 "묶어서 더 잡아내는" 것처럼 보였지만 사실은 모호함이 generalization 을 망가뜨리고 있었다. 단일 조건이 더 깨끗한 신호를 준다 (Occam's razor 적용 사례).
2. **Leakage 차단의 즉각적 효과** — `transform("mean")` (전체 평균) → `train rows 만의 quantile` 변경 만으로 val/test gap 이 거의 사라졌다. 작은 leakage 라도 generalization 을 측정 불능 상태로 만든다는 점 확인.
3. **Pipeline 순서 재구성의 부수효과** — split 을 label 생성 전에 수행하니 val 표본 1분기 추가. 의도하지 않았으나 metric 안정성에 긍정적. **순서가 신호에 영향** 을 준다는 점 학습.
4. **Subagent-driven-dev 의 적정 사용** — 작은 scope (T4 회귀 격리, T5 retrain) 는 controller inline 이 더 빠름. subagent 는 ≥ 30분 작업, 다중 파일 수정, TDD 사이클 필요한 task 에 적합. 무차별 dispatch 는 overhead.
5. **Sprint 의 batch commit 승인** — memory rule (commit 사전 확인) 을 sprint 단위로 한 번 일괄 승인 받아 끊김 없이 진행. auto mode + memory rule 의 충돌 해소 패턴.
