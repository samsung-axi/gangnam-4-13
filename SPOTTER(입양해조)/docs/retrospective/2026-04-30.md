# 2026-04-30 회고 — closure_risk 검증 layer fix + 정직한 retrain

## 요약

폐업 위험도 ML 모델(LightGBM + TCN ensemble)의 검증 layer를 random split → time-based holdout 3분할로 교체하고, 5 metric (AUC / PR-AUC / P@10 / R@10 / Brier) + Calibration plot을 도입했다. spec → plan → 7-task 서브에이전트 실행 → production retrain 까지 완료. **결과: 기존 val AUC 0.85는 사실상 temporal leakage 의 인플레였고, 정직한 test AUC 는 0.5142 (사실상 random)**. 모델 자체가 production-ready 가 아니라는 진단 확정.

---

## 1. 배경

기존 train.py 는 `train_test_split(random_state=42)` 로 random split 후 val AUC 0.85 를 보고했다. 그러나:
- 분기별 시계열(2021Q4 ~ 2024Q4) 데이터를 random 으로 섞으면 **미래 분기의 row 가 train 에 섞여 들어가는 temporal leakage** 가 발생.
- AUC 단일 지표만 보고했고, 양극불균형(pos_ratio≈0.27) 상황에서 PR-AUC / P@K / Calibration 이 빠져 있었다.
- "84% 정확도" 표기가 dashboard 에 production marker 로 쓰이고 있었으나, 검증 자체가 부정확했다.

**가설**: random split 으로 부풀려진 점수일 가능성 → time-based holdout 으로 교체 시 큰 폭의 하락이 예상된다.

## 3. 결과 — 정직한 metric

split: `train_quarters = 21Q4~22Q4 (5)`, `val_quarters = 23Q1~23Q3 (3)`, `test_quarters = 23Q4~24Q3 (4)`.

| 모델 | split | AUC | PR-AUC | P@10 | R@10 | Brier | pos_ratio |
|---|---|---|---|---|---|---|---|
| LGBM | val | 0.5355 | 0.2888 | 0.319 | 0.121 | 0.220 | 0.263 |
| LGBM | test | 0.5081 | 0.3293 | 0.290 | 0.090 | 0.251 | 0.323 |
| TCN | val | 0.5544 | 0.3123 | 0.362 | 0.138 | 0.194 | 0.262 |
| TCN | test | 0.5402 | 0.3645 | 0.403 | 0.125 | 0.218 | 0.323 |
| **Ensemble** | val | 0.5400 | 0.2943 | 0.340 | 0.129 | 0.200 | 0.264 |
| **Ensemble** | **test** | **0.5142** | **0.3305** | **0.339** | **0.105** | **0.226** | **0.323** |

ensemble weight: `w_lgbm=0.491, w_tcn=0.509`.

### 진단
- **Test AUC 0.5142** — random 보다 약간 위. 기존 random split 0.85 는 거의 전부 temporal leakage 인플레였다.
- **PR-AUC 0.33 vs baseline 0.32 (pos_ratio)** — 거의 베이스라인. 신호 미미.
- **P@10 = 0.339 vs 0.323** — 1.05× 베이스라인. top 10% 도 사실상 random sampling.
- **Brier 0.226** — 양호하지만 calibration 자체가 "거의 평균값으로 찍기" 결과일 가능성 높음 (calibration plot 참조).
- **결론**: production-ready 가 아니다. 84% 표기는 잘못된 marker 였다.

## 4. 다음 sprint 우선순위

`closure_risk` 의 본질 issue 는 검증이 아니라 **모델 자체** 였다는 점이 확인됨. brainstorm 단계에서 식별한 5 layer 중 E (검증) 만 손봤고, 진짜 병목은 다음 4 layer:

| Layer | 추정 비중 | 작업 |
|---|---|---|
| **C — label** | 매우 큼 | "분기 폐업 여부" binary label 자체의 정의 검토 (행정동 레벨 row 정의가 불균형 원인일 가능성) |
| **D — threshold** | 큼 | 단일 0.5 threshold 가 의미 있는지, top-K decision 로 전환 검토 |
| **B — feature** | 중 | 시계열 lag / rolling / exo (유동인구·매출) 추가 |
| **A — 구조** | 작음 | 단순 LGBM/TCN 앙상블이 적합한지, hierarchical / GNN 검토 |

권장: **C → D → B → A** 순. 각 layer 마다 별도 spec.

## 5. 배운 점

1. **"높은 AUC" 만으로 production marker 를 박지 마라.** validation 설계가 데이터 도메인(시계열) 에 맞는지부터 확인.
2. **PR-AUC + Calibration 은 무료다.** AUC 만 보고하면 imbalanced + 비교정 모델을 놓친다.
3. **Subagent two-stage review (spec → quality)** 는 비용 대비 효과 큼 — 6건의 HIGH 이슈가 잡혔다 (특히 NaN JSON, index mismatch, k>n 분모).
4. **Spec 명시 limitation 은 future spec 으로 보류** — T5 의 proba alignment trim 처럼 scope 안에서 깔끔하게 처리.
5. **Production retrain 직전에 numpy → native 변환을 깜빡했다.** `evaluate.py` 가 단독 함수 호출에서는 동작하지만, `train()` end-to-end 에서 numpy 타입이 끼면 `json.dump` 가 죽는다 → `_to_native()` 추가.
