# 프롬프트 시스템 구조 개선 보고서

## 1. 문제 진단: 단일 프롬프트 방식의 한계

초기에는 모든 지시사항, 규칙, 예시를 하나의 거대한 프롬프트에 담아 처리하였으나, 서비스가 고도화됨에 따라 다음과 같은 치명적인 문제들이 발생했습니다.

### 1-1. 인지 부하 및 주의력 분산 (Attention Limits)
*   **핵심 누락**: LLM(Large Language Model)의 입력 컨텍스트(Context Window)가 길어질수록, 모델이 프롬프트의 중간이나 방대한 지시사항 속에 숨겨진 핵심 규칙을 놓치거나 무시하는 현상이 발생했습니다.
*   **환각 현상(Hallucination) 증가**: 단계별로 명확히 구분되어야 할 정보들이 하나의 긴 텍스트에 섞이면서, 서로 다른 맥락의 정보가 충돌하여 잘못된 추론을 유도할 확률이 높아졌습니다.

### 1-2. 유지 보수의 어려움 (Maintenance Overhead)
*   **복잡도 증가**: 1,000줄이 넘어가는 단일 텍스트 파일에서 특정 로직이나 어조(Tone)를 수정하려면 전체를 훑어야 했으며, 수정 과정에서 관련 없는 부분을 건드려 예기치 않은 사이드 이펙트가 발생할 위험이 컸습니다.
*   **일관성 유지 실패**: 안전 수칙과 같은 공통 골자를 수정해야 할 때, 여러 버전의 프롬프트를 일일이 수정하다가 일부가 누락되는 휴먼 에러 발생 확률이 높았습니다.

### 1-3. 비효율적 비용 구조 (Cost Inefficiency)
*   **토큰 낭비**: 매 요청 시마다 전체 프롬프트를 전송해야 하므로, 불필요한 공통 컨텍스트까지 반복적으로 과금되는 구조였습니다. (Input Token 비용 증가)

### 1-4. 디버깅의 어려움 (Hard to Debug)
*   **원인 파악 불가**: 원하는 결과가 나오지 않았을 때, 거대한 프롬프트의 어느 구절이 모델의 판단을 흐리게 했는지 특정하기가 매우 어려웠습니다.

## 2. 해결 방안: 모듈화 및 동적 조립 (Modular & Dynamic Assembly)

이러한 문제를 해결하기 위해, 거대한 프롬프트를 기능 단위로 쪼개고 실행 시점에 필요한 부분만 조립하는 **'동적 프롬프트 아키텍처'**를 도입했습니다.

### 2-1. 프롬프트 모듈화 (Modularization)
프롬프트를 역할과 성격에 따라 세 가지 핵심 영역으로 분리하여 관리합니다.

1.  **공통 모듈 (Core Module)**
    *   모든 분석 단계에서 변하지 않는 불변의 법칙을 정의합니다.
    *   예: "아이의 안전이 최우선이다", "전문가 페르소나", "JSON 출력 형식 준수"
2.  **단계별 기준 (Step-specific Criteria)**
    *   각 발달 단계(월령 등)나 상황에 특화된 정밀 체크리스트와 판단 기준입니다.
    *   예: "생후 12개월 걷기 발달 체크리스트", "위험 상황 감지 기준"
3.  **추출 전용 (Extraction Instructions)**
    *   영상의 시각적 정보에서 객관적 사실(메타데이터)만을 건조하게 뽑아내기 위한 독립적인 지침입니다.
    *   주관적 해석을 배제하고 팩트만 수집하는 단계에 사용됩니다.

### 2-2. 동적 프롬프트 조립 (Dynamic Assembly)
애플리케이션은 요청 상황(아이의 월령, 분석 목적, 상황)에 맞춰 필요한 모듈들을 실시간으로 로딩하고 조립(Compile)하여 최종 프롬프트를 완성합니다.

*   **유연성(Flexibility)**: '공통 모듈' 하나만 수정하면, 이를 참조하는 수십 가지의 상황별 프롬프트에 즉시 변경 사항이 반영됩니다.
*   **정확성(Accuracy)**: 모델에게 현재 태스크에 딱 필요한 정보만 선별하여 제공하므로, 모델의 주의력(Attention)을 핵심 작업에 집중시킬 수 있습니다.

---

## 3. 개선 결과

*   **유지 보수성 향상**: 복잡도가 획기적으로 낮아져, 새로운 기능 추가나 기존 로직 수정이 매우 간편해졌습니다.
*   **확장성 확보**: 새로운 발달 단계나 분석 시나리오를 추가할 때, 기존 모듈을 레고 블록처럼 재사용할 수 있게 되었습니다.
*   **품질 안정화**: 명확히 분리된 책임을 가진 모듈들이 상호작용하며 분석 결과의 일관성과 정확도가 크게 개선되었습니다.
