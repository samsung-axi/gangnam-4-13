# 이슈 해결 보고서: FFmpeg 기반 HLS 스트리밍 CPU 폭주 원인 분석 및 최적화

**날짜:** 2025-12-15
**작성자:** Antigravity (AI Assistant)
**상태:** ✅ 해결됨

---

## 1. 🚨 문제 상황 (Symptoms)

로컬 Docker 환경에서 `ENABLE_HLS_STREAMING=true`로 HLS 스트리밍을 활성화했을 때, **`fastapi` 컨테이너의 CPU 사용량이 400% ~ 600%까지 치솟는 현상** 발생. 이는 시스템 전반의 성능 저하를 유발함.

## 2. 🔍 원인 분석 (Root Cause)

**핵심 원인: FFmpeg의 비효율적인 설정 및 파이프라인**

세 가지 주요 원인이 복합적으로 작용하여 CPU를 과도하게 점유했습니다.

1.  **FFmpeg의 비효율적인 파이프라인 (과도한 인코딩)**
    - 기존 로직은 HLS용과 아카이브용으로 각각 별도의 인코딩을 수행하거나, 불필요한 재인코딩이 발생했습니다.
    - 비디오 인코딩은 가장 무거운 작업인데, 이를 중복해서 수행하느라 막대한 리소스가 소모되었습니다.

2.  **FFmpeg의 실시간 처리 옵션(`-re`) 누락 (Critical)**
    - 업로드된 비디오 파일(VOD)을 소스로 사용하면서 FFmpeg에 `-re` (Read at native frame rate) 옵션을 주지 않았습니다.
    - 이로 인해 FFmpeg는 **"가능한 한 가장 빠른 속도"**로 변환을 시도했고, 10분짜리 영상을 수 초 만에 처리하려다 보니 CPU 코어를 100% 풀가동했습니다.

3.  **비동기 이벤트 루프 충돌 (Python/Asyncio 레벨)**
    - 백그라운드 스레드에서 아카이브 파일을 모니터링하고 후속 작업(S3 업로드 등)을 처리할 때, 이미 실행 중인 메인 이벤트 루프에 동기적으로 접근하려다 `RuntimeError: This event loop is already running`이 발생.
    - 이로 인한 예외 처리 및 재시도 루프가 반복되면서 리소스 낭비에 기여했습니다.

**참고**: HLS는 비디오 스트리밍 프로토콜(출력 형식)이며, CPU를 직접 사용하지 않습니다. CPU 폭주의 실제 원인은 **FFmpeg의 비디오 인코딩 작업**입니다.

## 3. 🛠️ 해결 방법 (Resolution)

`backend/app/services/live_monitoring/hls_stream_generator.py`를 대대적으로 최적화했습니다.

### 3.1 이중 인코딩 제거 (Double Encoding 제거)
- 기존에는 HLS용과 아카이브용으로 각각 별도의 인코딩을 수행했으나, 현재는 **HLS만 생성**하고 아카이브는 별도 프로세스로 처리합니다.
- HLS `.ts` 파일들을 주기적으로 모아서 `.mp4`로 변환하는 방식으로, 재인코딩 없이 `copy` 모드로 처리하여 CPU 부하를 최소화했습니다.

```python
# 현재 파이프라인 구조
[입력] -> [인코딩(libx264)] -> [HLS 출력 (.ts 파일들)]
                                    ↓ (10분마다)
                            [.ts 파일들 수집] -> [FFmpeg concat + copy] -> [아카이브 .mp4]
```

### 3.2 `-re` 옵션 추가 (The Game Changer)
이 문제 해결의 **가장 결정적인 핵심(Key Factor)**은 FFmpeg의 `-re` 옵션이었습니다.

#### 💡 기술적 배경 (Why it matters)
- **`-re` (Read at native frame rate)** 옵션은 FFmpeg에게 입력 파일을 **"영상의 실제 재생 속도(1x)"에 맞춰서 천천히 읽으라고 지시**하는 옵션입니다.
- **옵션 부재 시 (Transcoding Mode)**:
    - FFmpeg는 "가능한 한 빨리" 변환을 완료하려 합니다.
    - 마치 동영상 편집 프로그램 렌더링처럼, 10분짜리 영상을 10초 만에 처리하려 하므로 **CPU를 100% 풀가동(Max Load)**합니다.
    - 결과적으로 1초에 수백 프레임을 처리하게 되어 시스템이 폭주합니다.
- **옵션 적용 시 (Live Streaming Mode)**:
    - FFmpeg는 입력 영상의 메타데이터(FPS, Timebase)를 읽어, **현실 시간과 동기화**된 속도로만 패킷을 처리합니다.
    - 예를 들어 30fps 영상이라면, 정확히 1초에 30프레임만 처리하고 나머지는 대기(Sleep)합니다.
    - 이로 인해 CPU는 **"일하고 쉬고"를 반복**하게 되며, 사용량이 10%~20% 대로 급격히 안정화됩니다.

#### 🔄 타이밍 동기화 효과 (Synchronization)
이 옵션은 단순히 CPU만 낮추는 것이 아니라, 전체 **파이프라인의 시간 무결성**을 보장합니다.
- **Before**: 10분짜리 아카이브 파일(`archive_...mp4`)이 실제로는 10초 만에 생성됨 → "20분 전 영상"을 분석하려는 워커와 시간 축이 어긋남.
- **After**: 영상 생성 시간이 실제 흐르는 시간과 정확히 일치(1:1) → 워커가 참조하는 시점과 파일의 실제 내용이 완벽하게 동기화됨.

### 3.3 비동기 및 스레드 최적화
- **Async Fix**: 백그라운드 스레드에서 `asyncio.run_coroutine_threadsafe()`를 사용하여 메인 이벤트 루프에 작업을 안전하게 위임.
- **CPU Limit**: FFmpeg 실행 시 `-threads 2` 옵션을 추가하여, Docker 환경 내에서 사용할 수 있는 최대 스레드 수를 제한.

## 4. ✅ 결과 (Result)

- **CPU 사용량**: 600% → **10% ~ 20% 대**로 획기적 감소.
- **안정성**: 아카이브 파일이 정상적인 10분 간격으로 생성되며, 분석 스케줄러와 타이밍이 정확히 일치.
- **리소스 효율**: HLS만 생성하고 아카이브는 재인코딩 없이 처리하여 시스템 여유 자원 확보.

이로써 로컬 개발 환경에서도 쾌적하게 HLS 스트리밍과 AI 분석 기능을 테스트할 수 있게 되었습니다.

## 📝 주요 최적화 포인트 요약

### 1. FFmpeg `-re` 옵션
- **효과**: CPU 사용량 600% → 10-20%로 감소
- **원리**: 실시간 스트리밍 모드로 전환하여 프레임 처리 속도를 실제 재생 속도에 맞춤

### 2. 이중 인코딩 제거 (HLS 우선 생성 + 아카이브 후처리)
- **효과**: HLS만 생성하고 아카이브는 재인코딩 없이 `.ts` 파일들을 `copy` 모드로 합쳐서 CPU 부하 절감
- **구조**: HLS `.ts` 파일들을 주기적으로 모아서 `.mp4`로 변환 (재인코딩 없음)

### 3. 비동기 처리 최적화
- **방법**: `asyncio.run_coroutine_threadsafe()` 사용
- **효과**: 이벤트 루프 충돌 방지 및 안정성 향상

### 4. 스레드 제한
- **방법**: FFmpeg `-threads 2` 옵션
- **효과**: Docker 환경 내 CPU 코어 사용량 제한
