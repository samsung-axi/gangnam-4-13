## DailyCam 코드 리뷰 (2025-12-15)

**대상 범위**
- 백엔드: `backend/app` 전체 (FastAPI, DB, 스트리밍, 워커)
- 프론트엔드: `frontend/src` 전체 (라우팅, 페이지, HLS 플레이어 등)
- 인프라/문서: `docker-compose*.yml`, `docs/` 주요 아키텍처/트러블슈팅 문서

---

## 1. 아키텍처 & 구조

- **백엔드 (FastAPI + 모듈 구조)**
  - `app/main.py`에서 FastAPI 앱 생성 + CORS/세션 미들웨어 + startup 이벤트(결제 워커, HLS 자동 시작, 정리 스케줄러 등)를 한 곳에서 관리.
  - API는 기능별로 `app/api/<feature>/router.py`로 잘 분리되어 있고, 도메인별 서비스(`services/`), 모델(`models/`), DB 세션(`database/`) 계층도 비교적 명확함.
  - 라이브 스트리밍/아카이브/워커 등 실시간 성격이 강한 부분은 별도의 서비스/워커 클래스로 분리되어 있어 유지보수성이 나쁘지 않음.

- **프론트엔드 (React + Router)**
  - 진입점: `frontend/src/main.tsx` 에서 `BrowserRouter + App` 패턴으로 구성.
  - 페이지는 `src/pages/`로, 재사용 가능한 뷰/위젯은 `src/components/`, 비즈니스 단위 기능은 `src/features/`에 배치되어 있어, 구조가 직관적.
  - 타입 정의(`src/types`), API 래퍼(`src/lib/api.ts`), 공통 상수(`src/constants`) 등이 분리되어 있어 대규모 확장 시에도 방향성이 명확함.

- **문서 & 인프라**
  - `docs/backend`, `docs/deployment`, `docs/troubleshooting`, `docs/archive` 등으로 세분화되어, “설계 / 운영 / 트러블슈팅 / 히스토리”가 잘 나눠져 있음.
  - `docker-compose.yml`(로컬)와 `docker-compose.production.yml`, `docker-compose.streaming.yml`, `docker-compose.worker.yml` 이 역할별로 분리되어 있어, 서버 분리 아키텍처를 뒷받침.

**총평**: 전반적인 계층/폴더 구조는 **기능 단위로 잘 모듈화된 상태**이며, 스트리밍처럼 복잡한 부분도 서비스/워커/문서로 정리가 괜찮은 편입니다.

## 2. 백엔드 코드 품질 & 패턴

- **환경 변수 & 설정**
  - `.env`를 명시적으로 로드하고(`dotenv`), DB/S3/CORS 등 주요 설정을 환경 변수에 위임하는 패턴이 일관됨.
  - `ENVIRONMENT`, `ENABLE_HLS_STREAMING`, `WORKER_ID` 등으로 실행 모드를 분리해, 메인 서버/스트리밍 서버/워커를 하나의 코드베이스로 다루는 설계가 좋음.

- **DB 레이어**
  - `database/session.py`에서 `DATABASE_URL`을 조합하고, `SessionLocal`, `get_db()` 의존성 패턴을 사용해 FastAPI와 자연스럽게 연결.
  - `test_db_connection()` 유틸이 있어, startup 시 DB 상태를 확인하고 친절한 로그를 남김 (운영 트러블슈팅에 유리).
  - 모델은 도메인별로 나눠져 있으며, `Base.metadata.create_all()`를 startup에서 호출해 초기 배포 시 진입 장벽을 낮춤.

- **스트리밍 & 비디오 처리**
  - HLS 스트리밍은 `live_monitoring/hls_stream_generator.py` 에서 **단일 FFmpeg 프로세스 + 통합 파이프라인**으로 관리.
  - 분석 워커, 아카이브 업로드, S3 연동 등이 각 서비스 클래스로 나뉘어 있고, S3 비활성화 시 우회 경로를 고려하는 등 방어적 코딩이 이루어져 있음.
  - FFmpeg 경로 자동 탐지 및 Dockerfile에서의 설치, Windows용 auto-download 스크립트 등 **환경 간 차이**에 대한 고려가 잘 반영됨.

- **로깅 & 에러 처리**
  - S3/FFmpeg/워커 등에서 예외 처리 시 대부분 로그를 상세하게 남기고, 치명적이지 않은 경우 서비스는 계속 동작하도록 설계.
  - 다만, 일부 API 경로에서는 예외가 그대로 FastAPI까지 전파될 가능성이 있어, “사용자에게 보여지는 에러 메시지” 레벨에서는 추가 정리가 가능해 보임.

**백엔드 개선 제안**
- **도메인 서비스/리포지토리 패턴 정리**  
  - 일부 기능은 이미 서비스 클래스로 잘 분리되어 있지만, 일부 라우터는 아직 비즈니스 로직이 router 내부에 섞여 있을 수 있음.  
  - 장기적으로는 `api(입력/출력) - service(비즈니스) - repository(쿼리)` 레이어를 더 일관되게 가져가면 테스트/리팩터링이 용이해짐.
- **에러 응답 통일**  
  - 인증/결제/스트리밍 에러 응답 포맷을 공통 스키마로 묶고, 공통 예외 핸들러를 두면 프론트에서 처리 로직이 단순해질 수 있음.

## 3. 프론트엔드 코드 품질 & 패턴

- **상태 관리 & 라우팅**
  - React Router 기반 페이지 구조가 깔끔하고, 모니터링/대시보드/리포트/설정 등 도메인별 페이지가 분명히 나뉨.
  - 일부 상태는 `AppLayout` 또는 context를 통해 상위에서 관리(예: 전역 HLS 플레이어), 일부는 페이지 로컬 상태로 관리하는 하이브리드 패턴.

- **HLS 플레이어 & 스트리밍 UX**
  - `Monitoring.tsx`에서 hls.js를 직접 사용해 라이브 엣지, 버퍼 길이, 오류 복구 등 세세하게 튜닝한 부분이 있음.
  - LiveMonitoring 페이지는 전역 플레이어(AppLayout)와의 연동을 전제로 설계되어 있었고, 최근 병합에서 HLS 컴포넌트를 직접 import 하는 구조와 충돌했던 이력이 있음.

- **타입/유틸/구조**
  - API 응답 타입, 도메인 타입이 `src/types`에 정의되어 있고, `api.ts` 래퍼와 함께 사용되는 패턴은 유지보수에 유리.
  - `constants`, `utils`, `features` 폴더가 역할별로 나뉘어 있어, 공통 코드 재사용이 가능한 상태.

**프론트엔드 개선 제안**
- **전역 플레이어 vs 페이지별 플레이어 아키텍처 확정**
  - 현재 문서/코드 흐름상 “전역 HLS 플레이어(AppLayout 상단) + LiveMonitoring은 컨트롤/뷰만” 이라는 방향이 맞지만, 병합 과정에서 페이지 내부 플레이어 시도가 섞임.
  - HLS 재생 책임을 어디에 둘지(전역 vs 페이지) 팀 레벨에서 한 번 더 확정하고, 그에 맞게 `LiveMonitoring.tsx`/`Monitoring.tsx`/공통 HLS 컴포넌트 구조를 정리하는 게 좋음.
- **에러/로딩 UX 통일**
  - API 호출/스트리밍 실패 시 사용자에게 보여주는 메시지/스피너/재시도 버튼 등을 공통 컴포넌트로 빼면 UX 일관성이 올라감.

## 4. 스트리밍 & 성능 관점 리뷰

- **강점**
  - FFmpeg 기반 HLS 인코딩 + hls.js 튜닝 값(버퍼, 라이브 엣지, 재연결) 조합으로 저지연/안정성을 동시에 고려.
  - S3를 “켜도 되고, 꺼도 되도록” 설계 (`is_enabled()` 체크), 로컬 파일 캐시를 기본으로 두어 S3 의존성을 줄인 점이 좋음.
  - `docs/troubleshooting/STREAMING_PERFORMANCE_GUIDE(스트리밍_성능_최적화).md` 등 문서가 매우 상세해, 운영 시 튜닝 포인트를 개발자가 빠르게 따라갈 수 있음.

- **리스크/주의점**
  - FFmpeg 프로세스/워커/스트림 관리가 복잡하기 때문에, 일부 경로에서 race condition(파일 닫히기 전 접근, 삭제/생성 타이밍) 이슈가 다시 나타날 수 있음.
  - 라이브 스트림/아카이브/워커가 모두 DB와 파일 시스템, S3를 공유하므로, 환경별(로컬/스테이징/프로덕션) 설정이 조금만 달라도 예상치 못한 동작이 나올 수 있음.

**개선 제안**
- **스트리밍 상태 대시보드/API**  
  - 현재 로그 기반으로 상태를 파악하는 구조인데, 간단한 “스트리밍 상태 조회 API + 프론트 대시보드”가 있으면 운영자가 FFmpeg/HLS 상태를 눈으로 볼 수 있어 디버깅이 쉬움.
- **헬스체크/알람**  
  - FFmpeg 프로세스 종료, 아카이브 업로드 실패, S3 다운로드 실패 등의 중요 이벤트를 별도 메트릭/알람 채널(예: Slack, 이메일)로 연결할 수 있으면 프로덕션 안정성이 올라감.

## 5. 배포 & 운영 관점

- **Docker & 구성**
  - 로컬/프로덕션/스트리밍/워커 compose 파일이 분리되어 있으며, 환경 변수 템플릿(`env.production.template`)이 잘 정리되어 있어 초기 배포 플로우가 명확함.
  - Nginx, FastAPI, MySQL, (옵션) 워커가 명확한 역할 분담을 갖고 있음.

- **환경 변수 설계**
  - S3, CORS, OAuth, 결제, VLM 등 외부 서비스 설정을 전부 ENV로 몰아놓은 점은 12-factor 관점에서 좋음.
  - 다만, 프로덕션에서 필수로 필요하지만 기본값이 비어있는 변수들(예: S3 관련)이 있으므로, 실제 배포 시 체크리스트/CI 검증을 두면 좋음.

**개선 제안**
- **환경 변수 검증 레이어**  
  - 애플리케이션 startup 시 “필수 ENV가 없으면 명확한 에러 메시지를 띄우고 종료”하는 검증기를 하나 두면, 잘못된 설정으로 반쯤 동작하는 상황을 줄일 수 있음.
- **테스트/스테이징 프로파일**  
  - 현재 ENV 기반으로 모드를 나누지만, `ENVIRONMENT=test/staging/production` 별로 로깅 레벨/FFmpeg/HLS/S3 옵션을 프리셋으로 관리하는 것도 고려해 볼 만함.

## 6. 종합 평가 & 다음 단계 제안

- **종합 평가**
  - 기능 분리, 문서화, 스트리밍/AI/결제 등 복잡한 도메인을 한 코드베이스에서 다루면서도, 구조적으로 크게 어긋나는 부분은 없고 **실서비스 수준의 설계**를 갖추고 있음.
  - 특히 스트리밍/FFmpeg/HLS/S3/워커 관련 문서화가 잘 되어 있어, 향후 새로운 팀원이 합류해도 학습 곡선이 상대적으로 낮을 것으로 보임.

- **단기적으로 추천하는 작업**
  1. **LiveMonitoring / Monitoring / 전역 HLS 플레이어 구조 확정 및 정리**
  2. **핵심 API(인증/결제/스트리밍 시작/중지)에 대한 에러 응답 포맷 통일**
  3. **스트리밍 상태/워커 상태를 볼 수 있는 간단한 내부 대시보드 추가**

- **중기적인 방향**
  - 서비스/리포지토리 레이어 정리, 공통 에러/로그 포맷 도입, 헬스체크/메트릭 연동 등으로 “운영 관점의 안정성”을 더 다듬으면 좋겠습니다.
