# 스트레스 테스트 & 시스템 아키텍처 트래픽 요구사항 정의

## 1. 왜 스트레스 테스트를 해야 하는가?

### 1.1 병목(Bottleneck) 발견
- **어디다있소**의 핵심 파이프라인: STT → Intent 분석 → Hybrid Search → LLM Reranking
- 각 단계의 처리 시간이 다르며, LLM 호출(Gemini API)이 가장 큰 병목
- 실제 동시 접속자가 발생하면 어느 구간에서 지연이 발생하는지 **측정**해야 함

### 1.2 한계치(Capacity Limit) 파악
- Lightsail Micro 인스턴스 (1 vCPU, 512MB RAM): **QPM 상한선**이 얼마인지?
- 예상: 10~30 QPM (LLM 호출 포함 시), 100+ QPM (캐시 활용 시)
- 측정 없이 설계하면 **과소/과대 인프라 투자** 발생

### 1.3 SLA(Service Level Agreement) 기준 설정
- 매장 키오스크 서비스에서 사용자가 허용하는 대기시간: **최대 3~5초**
- P95 응답시간이 이 기준을 넘으면 UX 저하 → 사용 포기
- 스트레스 테스트로 P95/P99 지표를 확보해야 SLA 약속 가능

### 1.4 스케일링(Scaling) 판단 근거
- 테스트 결과를 바탕으로:
  - Scale(인스턴스 수) 증가 필요?
  - Power(CPU/RAM) 업그레이드 필요?
  - 캐시 레이어(Redis) 도입 필요?
  - API 호출 최적화(배치 처리) 필요?

### 1.5 장애 대비(Resilience)
- 예상 트래픽의 **2~3배** 부하에서 시스템이 어떻게 동작하는지 확인
- Graceful degradation: 과부하 시 에러 대신 "잠시 후 다시 시도" 안내 가능

## 2. 왜 시스템 아키텍처 설계 시 트래픽 요구사항을 정의하는가?

### 2.1 비용 최적화

| 시나리오 | 일 요청수 | 필요 인프라 | 월 비용 |
|---------|----------|------------|--------|
| 소규모 매장 1개 | ~500 | Lightsail Micro | ~$7 |
| 중규모 매장 5개 | ~5,000 | Lightsail Small x2 | ~$50 |
| 대규모 체인 50개 | ~100,000 | ALB + ECS Fargate | ~$200+ |

→ 트래픽 없이 설계하면 **과소 설계**(장애) 또는 **과대 설계**(비용 낭비)

### 2.2 아키텍처 분리 근거
- **낮은 트래픽** (< 30 QPM): FE + BE 단일 서버 가능
- **중간 트래픽** (30~100 QPM): FE/BE 분리, 캐시 도입
- **높은 트래픽** (100+ QPM): ALB, Auto Scaling, CDN 필요

### 2.3 기술 스택 선택 근거
- 동시 접속 10명 이하: 단일 FastAPI 인스턴스 충분
- 동시 접속 50명 이상: Worker 수 증가 + 비동기 처리 최적화
- 동시 접속 100명 이상: Kubernetes/ECS + 로드밸런서

### 2.4 이해관계자 소통
- "하루 10,000건의 검색 요청 처리 가능" — 구체적 수치
- "피크 시간(점심/저녁) P95 응답시간 2초 이내" — 측정 가능한 목표
- 비기술 이해관계자에게도 시스템 성능을 명확히 전달

### 2.5 단계적 확장 계획
```
Phase 1 (PoC):  Lightsail Micro, 단일 서버     → 검증
Phase 2 (MVP):  Lightsail Small, FE/BE 분리    → 소규모 운영
Phase 3 (Prod): ALB + ECS, Redis Cache         → 다중 매장
Phase 4 (Scale): CDN + Auto Scaling + 모니터링  → 전국 확장
```

## 3. 어디다있소의 트래픽 요구사항 정의

### 3.1 예상 사용 시나리오

| 항목 | 값 | 근거 |
|------|-----|------|
| 매장 운영시간 | 10시간/일 | 10:00~22:00 |
| 키오스크 대수 | 2대/매장 | B1, B2 각 1대 |
| 검색 빈도 | 2~3회/분 (피크) | 주말/점심시간 관찰 |
| 평균 검색 | 1회/분 | 평일 평균 |
| 일 총 요청 | ~600회 | 10시간 × 60분 × 1회 |
| 피크 QPM | ~6 QPM | 2대 × 3회 |

### 3.2 성능 목표 (SLA)

| 지표 | 목표값 |
|------|--------|
| P95 응답시간 | < 3,000ms |
| P99 응답시간 | < 5,000ms |
| 가용성 | > 99% |
| 동시 접속 | 5명 이상 |
| QPM | 30+ (여유 5배) |

### 3.3 테스트 계획

```bash
# 1단계: 기본 성능 확인 (10 QPM)
python tests/stress_test_qpm.py --url <lightsail-url> --qpm 10 --duration 60

# 2단계: 목표 QPM (30 QPM)
python tests/stress_test_qpm.py --url <lightsail-url> --qpm 30 --duration 120

# 3단계: 한계 테스트 (60+ QPM)
python tests/stress_test_qpm.py --url <lightsail-url> --qpm 60 --duration 60
```
