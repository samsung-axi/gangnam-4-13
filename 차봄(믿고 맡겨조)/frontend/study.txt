
[AI 차량 예지 정비 시스템 - 발표 가이드]

================================================================================
1. 프로젝트 개요
================================================================================

프로젝트명: AI 차량 예지 정비 시스템 (AI-5)
핵심 가치: 차량의 미래를 예측하고, 고장을 사전에 방지하며, 유지비용을 투명하게 관리

주요 기능
1. OBD-II 기반 실시간 주행 분석: 실시간 데이터 수집 및 안전 운전 점수 산출
2. AI 멀티모달 진단: 시각(Vision), 청각(Audio), 대화형(Chat) 진단 제공
3. 스마트 소모품 관리: 주행 거리 기반 자동 카운트다운 및 영수증 OCR 인식 (New)
4. RAG 기반 예지 정비 챗봇: 차량 매뉴얼 + 정비 이력 기반 맞춤형 상담
5. Smartcar API 통합: 커넥티드 카 데이터 연동 (New)

시스템 구성
```
┌─────────────┐         ┌──────────────┐        ┌─────────────┐
│   Mobile    │◄───────►│   Backend    │◄──────►│  AI Server  │
│  (React     │  HTTPS  │ (Spring Boot)│  REST  │  (FastAPI)  │
│   Native)   │  +JWT   │              │        │             │
└─────────────┘         └──────────────┘        └─────────────┘
      ▲                        ▲                      ▲
      │                        │                      │
      │                        ▼                      │
  OBD-II              ┌──────────────┐                │
  Bluetooth           │  PostgreSQL  │                │
                      │   +pgvector  │                │
                      │  TimescaleDB │                │
                      └──────────────┘                │
                               ▲                      │
                               │                      │
                               └───────────────────────┘
                                   Vector Search
```

기술 철학
- Hybrid AI: 로컬 모델(속도) + 클라우드 LLM(정확도)
- Offline-First: 네트워크 단절에도 데이터 유실 방지
- Cost-Efficient: Router 모델로 불필요한 API 호출 필터링

2. 핵심 데이터 플로우
================================================================================

[주행 데이터 수집 → 분석 → 알림]

1. 앱 실행 시 OBD 자동 연결 → Bluetooth 스캔 및 페어링
2. Foreground Service가 Speed > 0 감지 시 자동 주행 시작
3. 1초 간격 데이터 수집 (Speed, RPM, Coolant, Load 등)
4. 주행 종료 (Speed = 0 && RPM = 0이 10분 지속)
   → 주행 통계 계산 (평균 속도, 안전 점수, 급가속 횟수 등)
   → 소모품 마모 가중치 업데이트
   → 잔존 수명 < 30% 시 FCM 푸시 알림

[멀티모달 AI 진단]

Vision (시각 진단)
1. 사용자가 엔진룸/계기판 촬영
2. YOLOv11m-CLS → 장면 분류 (4종: 엔진룸/계기판/외관/타이어)
3. YOLOv11 Specialist → 부품 탐지 (배터리, 오일캡 등 50종+)
4. GPT-4o Vision → 결함 해석 ("오일 누유 의심", "벨트 균열")

Audio (청각 진단)
1. 사용자가 엔진 소음 녹음
2. Mel-Spectrogram 변환 → AST Model 분류
3. 이상음 탐지 (노킹, 벨트 소음 등)
4. GPT-4o로 2차 해석 및 권고사항 제시

RAG 챗봇
1. 사용자 질문 → Embedding 벡터 변환
2. pgvector로 차량 매뉴얼 검색 (메타데이터 필터링)
3. 검색된 컨텍스트 + 질문 → GPT-4o 답변 생성
4. 할루시네이션 방지 (검색된 매뉴얼 기반 답변)

4. 기술 스택
================================================================================

A. Frontend (Mobile App - React Native)
--------------------------------------------------------------------------------
- React Native 0.81 + Expo SDK 54
  - 선정 이유: 크로스 플랫폼 개발로 iOS/Android 동시 지원, 빠른 개발 속도
  - Hot Reload로 즉각적인 피드백, Native 모듈과의 확장성 결합

- TypeScript 5.x
  - Strict Type Checking으로 런타임 에러 사전 차단
  - 대규모 코드베이스에서도 안전한 리팩토링 가능

- Zustand (상태 관리)
  - 선정 이유: Redux 대비 경량화, OBD 센서 데이터처럼 고빈도 업데이트에 최적화
  - 단방향 데이터 흐름으로 예측 가능한 상태 관리
  - 주요 Store: useVehicleStore, useObdStore, useAiDiagnosisStore

- Offline-First Architecture
  - AsyncStorage 로컬 저장 → 네트워크 복구 시 자동 동기화
  - 타임스탬프 기반 충돌 해결 로직

- Dual-Mode Bluetooth
  - BLE Manager: 최신 BLE 4.0+ OBD 스캐너 (저전력 모드)
  - Bluetooth Classic: 구형 ELM327 호환 (SPP 프로토콜)
  - 우선순위 큐: 주행 시작/종료 명령 최우선 처리

- Multimedia Processing
  - expo-camera: 이미지 quality 0.8 압축, EXIF 메타데이터 기록
  - expo-av: 48kHz HIGH_QUALITY 샘플링, 실시간 파형 시각화
  - Google Cloud Vision (OCR): 영수증 텍스트 인식 및 주행 거리 자동 추출

- Foreground Service
  - Android Foreground Service로 OS 종료 방지
  - Doze 모드에서도 데이터 수집 보장

- Axios Interceptors
  - Request: JWT Access Token 자동 헤더 추가
  - Response: 401 시 Refresh Token으로 자동 재발급, 네트워크 에러 시 재시도 (최대 3회)


B. Backend (API Server - Spring Boot)
--------------------------------------------------------------------------------
- Spring Boot 3.5.9 (Java 17 LTS)
  - 선정 이유: 엔터프라이즈급 안정성, 대규모 트래픽 처리 검증됨
  - Auto-Configuration으로 빠른 개발, Embedded Tomcat으로 독립 실행

- Java 17 LTS Features
  - Records: DTO 보일러플레이트 제거
  - Pattern Matching: instanceof 체크와 캐스팅 동시 처리
  - Sealed Classes: 타입 안전성 강화

- Spring Data JPA + QueryDSL
  - JPA: Repository 인터페이스만 정의하면 CRUD 자동 구현
  - QueryDSL: 컴파일 타임 타입 체크 가능한 동적 쿼리 빌더
  - @EntityGraph로 N+1 문제 방지

- PostgreSQL + TimescaleDB
  - 시계열 데이터 최적화: OBD 로그(초당 수백만 건) 효율 저장
  - Hypertable: 시간 기반 자동 파티셔닝
  - Continuous Aggregates: 일/주/월별 통계 자동 계산

- Spring Security + JWT
  - Stateless 인증으로 확장성 확보
  - Access Token 15분 / Refresh Token 7일
  - Token Rotation으로 보안 강화

- BCrypt Password Hashing
  - Salt 자동 추가로 Rainbow Table 공격 방지
  - Work Factor로 미래 컴퓨팅 파워 증가 대응

- RabbitMQ
  - AI 분석 같은 무거운 작업 비동기 처리
  - Message Durability + Dead Letter Queue로 신뢰성 확보

- Integration Services
  - AWS S3: 미디어 파일 저장 (Presigned URL 보안 적용)
  - Firebase FCM: 실시간 푸시 알림 (소모품 교체, 진단 완료)
  - Google Cloud Vision: 영수증 OCR 기반 자동 차계부 입력
  - Smartcar API: 커넥티드 카 데이터 연동 (테슬라, 현대/기아 등 지원)

- Testing
  - JUnit 5 (Parameterized Tests, Nested Tests)
  - MockMvc (통합 테스트)
  - Test Containers (실제 PostgreSQL 컨테이너 사용)

8. 발표 예상 질문 & 답변 (Top 40)
================================================================================

[프로젝트 기획 및 차별점]
Q1. 시중에 있는 마이클, 카닥 같은 차량 관리 앱과 무엇이 다른가요? (경쟁력)
A: 기존 앱들은 소모품 교체 주기 알림이나 정비소 예약 등 '관리'에 초점이 맞춰져 있습니다. 저희 AI-5는 OBD-II 데이터와 멀티모달 AI(이미지, 소리)를 결합하여 전문 장비 없이도 사용자 스스로 차량의 내/외부 결함을 '진단'하고, 예지 정비(Predictive Maintenance)까지 가능하다는 점이 가장 큰 차별점입니다.

Q2. 왜 하필 '예지 정비'를 주제로 선정했나요? (기획 배경)
A: 차량 고장은 예고 없이 찾아오지만, 대부분 전조증상이 있습니다. 초보 운전자는 이를 인지하기 어려워, 나중에 큰 수리비를 지출하게 됩니다. 저희는 누구나 쉽게 스마트폰 하나로 전문가 수준의 진단을 받고, 사전에 위험을 감지하여 비용과 안전을 모두 지키는 서비스를 만들고자 했습니다.

[OBD-II 및 데이터]
Q3. OBD-II 스캐너가 없는 사용자는 이 앱을 못 쓰나요? (진입 장벽)
A: 아닙니다. OBD 스캐너가 없더라도 Smartcar API 연동을 통해 커넥티드 카(현대, 기아, 테슬라 등) 데이터를 불러와 기본 점검이 가능합니다. 또한 영수증 OCR을 통한 차계부 관리, AI 소리/사진 진단 기능은 하드웨어 없이도 독립적으로 사용할 수 있습니다.

Q4. 데이터 수집 중 블루투스 연결이 끊기거나 터널을 지날 때는 어떻게 되나요? (안정성)
A: 'Offline-First' 아키텍처를 적용했습니다. 연결이 끊기면 로컬 저장소(SQLite)에 데이터를 임시 저장하고, 우선순위 큐(Priority Queue) 시스템이 네트워크/연결 복구 시 누락된 데이터를 순차적으로 서버에 전송하여 데이터 정합성을 보장합니다.

Q5. 실시간 데이터라고 했는데, 지연(Latency)은 없나요? (성능)
A: OBD 데이터는 초당 1회 수집되며, 모바일 앱에서 경량화(JSON)하여 전송합니다. 백엔드에서는 RabbitMQ 메시지 큐 비동기 처리를 도입하여 대량의 데이터 유입에도 사용자 경험(UX)에 영향을 주지 않도록 설계했습니다.

[AI 모델 및 정확도]
Q6. AI 진단의 정확도는 얼마나 신뢰할 수 있나요? (신뢰성)
A: 자체 테스트 결과 YOLOv11 기반 시각 진단은 95% 이상, AST 기반 청각 진단은 90% 이상의 정확도를 보였습니다. 특히 오분류를 줄이기 위해 1차로 MobileNet(Router)이 유효한 데이터인지 검증하고, 2차로 정밀 모델이 분석하는 2단계 검증 시스템을 구축했습니다.

Q7. 엔진 소리만으로 고장을 알 수 있나요? 주변 소음에 취약하지 않나요? (기술적 한계)
A: 엔진 소음은 특유의 주파수 패턴(노킹, 벨트 슬립 등)이 있습니다. 저희는 Mel-Spectrogram 변환을 통해 소리를 시각화된 패턴으로 분석하며, 학습 시 다양한 배경 소음(노이즈)을 포함시켜 강건성(Robustness)을 확보했습니다.

Q8. GPT-4o를 쓰면 비용이 많이 들 텐데, 수익 모델은 무엇인가요? (비즈니스)
A: 모든 요청에 GPT를 쓰지 않습니다. 단순 분류는 로컬 모델이 처리하고, 복잡한 증상 해석에만 GPT를 호출합니다. 수익 모델로는 정비소 연계 수수료, 프리미엄 리포트 구독(상세 분석), 그리고 자동차 보험사와의 데이터 제휴(UBI) 등을 고려하고 있습니다.

Q9. RAG(검색 증강 생성)는 왜 필요한가요? 그냥 GPT한테 물어보면 안 되나요? (기술)
A: 일반 LLM은 특정 차종의 매뉴얼이나 최신 리콜 정보를 모릅니다. 저희는 제조사 공식 매뉴얼을 Vector DB에 구축하여, AI가 반드시 매뉴얼에 근거한 답변만 하도록 하여 '할루시네이션(거짓 정보 생성)'을 원천 차단했습니다.

Q10. 이미지를 찍으면 개인정보(번호판 등) 유출 우려는 없나요? (보안)
A: 네, 저희는 이미지 업로드 직후 OpenCV를 통해 번호판이나 사람 얼굴 같은 민감 정보를 자동으로 마스킹(Blurring) 처리한 후 서버에 저장합니다. 또한 전송 구간은 HTTPS로 암호화되며 S3 버킷 접근 권한도 엄격히 제어됩니다.

[서비스 기능]
Q11. '안전 운전 점수'는 보험료 할인 같은데 활용할 수 있나요? (확장성)
A: 네, 급가속, 급감속 데이터를 기반으로 산출된 점수는 T-map 점수처럼 활용 가능합니다. 향후 API 형태로 보험사에 데이터를 제공하여 '안전 운전 특약 할인' 등의 연계 서비스를 기획하고 있습니다.

Q12. 소모품 수명은 단순히 주행거리로만 계산하나요? (알고리즘)
A: 아닙니다. 제조사 권장 주행거리 외에도 '가혹 주행 조건(급가속, 공회전, 고부하 운전)' 감지 시 마모도에 가중치를 부여하는 'Wear Factor' 알고리즘을 적용하여, 실제 차량 상태에 더 가까운 교체 주기를 안내합니다.

Q13. 영수증 OCR 인식이 잘 안될 때는 어떻게 하나요? (UX)
A: Google Cloud Vision API를 사용하여 높은 인식률을 자랑하지만, 구겨지거나 흐릿한 영수증을 대비해 사용자가 인식된 결과를 직접 수정할 수 있는 UI를 제공합니다. 또한 정규식(Regex) 후처리를 통해 날짜, 금액 데이터의 정확도를 높였습니다.

[아키텍처 및 개발]
Q14. 왜 React Native를 선택했나요? (기술 선정)
A: iOS와 Android를 동시에 지원해야 했고, 하드웨어(카메라, 블루투스) 제어가 중요했습니다. React Native는 크로스 플랫폼이면서도 Native Module 접근이 용이하고, Expo의 방대한 생태계를 활용하여 개발 속도를 획기적으로 단축할 수 있었습니다.

Q15. 백엔드에서 Spring Boot와 FastAPI를 나눈 이유는 무엇인가요? (아키텍처)
A: '역할의 분리'입니다. Spring Boot는 사용자 인증, 데이터 CRUD, 결제 등 안정적인 비즈니스 로직 처리에 강점이 있고, FastAPI는 Python 기반의 풍부한 AI 라이브러리(PyTorch, YOLO) 활용과 비동기 처리에 유리합니다. 각자 잘하는 영역에 집중하여 시스템 전체의 안정성과 성능을 모두 잡았습니다.

Q16. 데이터가 많아지면 DB가 느려질 텐데 대비책은 있나요? (확장성)
A: 시계열 데이터(OBD 로그)는 TimescaleDB(또는 파티셔닝) 적용을 고려하여 설계했고, 자주 조회되는 진단 이력이나 캐시 데이터는 Redis를 활용해 DB 부하를 줄였습니다. 장기적으로는 DB 샤딩이나 Read Replica 도입도 가능한 구조입니다.

Q17. 배터리 소모가 심하지 않나요? 백그라운드에서 계속 돌면... (최적화)
A: 가속도 센서를 활용해 차량이 멈춰있을 때는 데이터 수집 주기를 늘리는(Sleep Mode) 로직을 적용했습니다. 또한 블루투스 통신도 저전력(BLE)을 우선 사용하며, 불필요한 연산을 최소화하여 배터리 효율을 최적화했습니다.

[운영 및 기타]
Q18. 전기차(EV) 지원 계획은 있나요?
A: 현재는 내연기관 차량 중심이지만, 전기차 전용 OBD PID(배터리 전압, 셀 온도 등)를 분석 중에 있습니다. 구조적으로 PID 매핑만 추가하면 되므로 향후 업데이트를 통해 지원할 예정입니다.

Q19. 이 서비스를 실제 정비소에서 사용할 수 있을까요? (B2B)
A: 네, 정비소용 '프로 버전'을 통해 고객 차량의 사전 진단 리포트를 받아볼 수 있다면, 정비사는 문진 시간을 줄이고 정확한 정비가 가능해집니다. B2B 솔루션으로의 확장 가능성도 열려 있습니다.

Q20. 팀원 간 협업 시 가장 힘들었던 점과 해결 방법은? (협업)
A: AI 서버(Python)와 백엔드(Java) 간의 데이터 규격(Protocol)을 맞추는 것이 어려웠습니다. 이를 해결하기 위해 Swagger(OpenAPI)로 명세를 명확히 하고, DTO(Data Transfer Object)를 공유하며 지속적인 소통과 통합 테스트로 간극을 좁혔습니다.

[기술 심화 - 백엔드 & 인프라]
Q21. PostgreSQL + pgvector 사용 시 인덱싱은 어떻게 처리했나요? (성능)
A: 벡터 검색 속도 향상을 위해 HNSW(Hierarchical Navigable Small World) 인덱스를 사용했습니다.
IVFFlat 방식보다 빌드 시간은 길지만, 검색 속도와 정확도(Recall)가 훨씬 뛰어나 실시간 챗봇 응답에 적합하다고 판단했습니다.

Q22. Redis 캐시 전략(Cache Strategy)은 무엇인가요?
A: 'Look Aside' 패턴을 적용했습니다. 데이터를 찾을 때 Redis를 먼저 조회하고, 없으면 DB에서 가져와 Redis에 적재합니다.
또한, 모든 키에 TTL(Time-To-Live)을 설정하여 메모리 누수를 방지하고, Cache Stampede 현상을 막기 위해 만료 시간을 랜덤하게 분산시켰습니다.

Q23. JPA 사용 시 N+1 문제는 어떻게 해결했나요? (최적화)
A: 연관된 엔티티를 조회할 때 성능 저하가 발생하지 않도록 Fetch Join과 @EntityGraph를 적극 활용했습니다.
예를 들어, 차량 정보 조회 시 소모품 목록(One-to-Many)을 한 번의 쿼리로 가져오도록 최적화하여 쿼리 발생 횟수를 획기적으로 줄였습니다.

Q24. S3 Presigned URL을 쓴 구체적인 이유는? (보안)
A: 서버를 거쳐 파일을 업로드하면 서버 리소스(CPU/메모리)가 낭비됩니다.
Presigned URL을 사용하면 클라이언트가 AWS S3에 직접 업로드하되, 서버가 발급한 짧은 유효기간(약 5분)을 가진 URL로만 접근 가능하므로 '서버 부하 분산'과 '보안' 두 마리 토끼를 잡을 수 있습니다.

[기술 심화 - AI & 데이터]
Q25. YOLO 학습 데이터 부족 문제는 어떻게 해결했나요? (Data Augmentation)
A: 직접 촬영한 데이터는 한계가 있어, Albumentations 라이브러리를 사용해 회전, 밝기 조절, 블러링 등 다양한 증강 기법을 적용했습니다. 이를 통해 데이터셋을 3배 이상 증량하고, 모델의 일반화(Generalization) 성능을 높였습니다.

Q26. 오디오 데이터 전처리 과정이 궁금합니다. (Preprocessing)
A: 입력된 오디오를 5초 단위로 자른(Chunking) 후, Librosa를 사용해 128-band Mel-Spectrogram으로 변환했습니다.
또한, 인간의 청각 특성을 반영한 Log-Scale 변환을 거쳐 소리의 특징을 시각적 패턴으로 명확히 추출했습니다.

Q27. RAG에서 검색된 문서가 너무 많으면 어떻게 하나요? (Reranking)
A: 벡터 검색 결과 상위 10개를 1차로 뽑고, Cross-Encoder 모델을 사용해 질문과의 연관성을 다시 정밀 채점(Reranking)하여 최상위 3개만 LLM에 입력으로 전달합니다. 이를 통해 토큰 비용을 줄이고 답변 정확도를 높였습니다.

[기술 심화 - 프론트엔드]
Q28. 앱 용량과 로딩 속도 최적화는? (Performance)
A: Hermes Engine을 활성화하여 JS 바이트코드 로딩 속도를 개선했고, 불필요한 라이브러리는 제거(Tree Shaking)했습니다.
이미지는 WebP 포맷을 사용하여 용량을 30% 이상 줄였으며, React.memo와 useMemo를 활용해 불필요한 리렌더링을 방지했습니다.

Q29. 오프라인 동기화 시 데이터 충돌(Conflict)이 발생하면?
A: 'Last Write Wins(최신 데이터 우선)' 정책을 기본으로 하되, 중요한 주행 로그는 'Append Only(추가만 가능)' 방식으로 설계하여 데이터 덮어쓰기로 인한 유실을 방지했습니다.

Q30. React Native의 New Architecture(Fabric)를 도입했나요?
A: 현재는 안정성을 위해 기존 아키텍처를 사용 중이지만, 향후 JSI(JavaScript Interface)를 활용한 동기적 함수 호출로 Bridge 오버헤드를 줄이기 위해 마이그레이션을 검토 중입니다. 특히 실시간 OBD 데이터 처리에 큰 이점이 있을 것으로 예상합니다.

[운영 및 비즈니스 확장]
Q31. 개인정보 처리방침과 보관 기간은? (GDPR/Compliance)
A: 회원 탈퇴 시 즉시 파기(Hard Delete)를 원칙으로 하며, 주행 데이터는 통계 분석을 위해 비식별화(De-identification) 후 보관합니다. S3에 저장된 미디어 파일은 수명 주기 정책(Lifecycle Policy)에 따라 30일 후 자동으로 Glacier로 이동하거나 삭제됩니다.

Q32. 초기 사용자 확보(Go-to-Market) 전략은?
A: 자동차 동호회나 커뮤니티를 타겟으로 '내 차 소리 진단 챌린지' 같은 바이럴 마케팅을 기획하고 있습니다. 또한, 기본적인 OBD 스캐너를 판촉물로 제공하여 초기 진입 장벽을 낮추는 전략도 고려 중입니다.

Q33. 차량 정비소와의 제휴 모델은 구체적으로?
A: 사용자가 진단 리포트를 가지고 제휴 정비소를 방문하면 공임비 할인을 제공하고, 앱 내에서 정비소 예약/결제 시 플랫폼 수수료를 받는 수익 모델을 구상하고 있습니다.

Q34. 향후 로드맵(Next Step)은 무엇인가요?
A: 1단계는 '진단'이지만, 2단계는 '부품 구매'와 '정비 예약'까지 연결하는 O2O 플랫폼입니다. 더 나아가 수집된 방대한 차량 데이터를 활용해 중고차 가치 산정이나 보험 상품 개발 등 데이터 비즈니스로 확장할 계획입니다.

Q35. 개발 중 가장 어려웠던 버그는? (Troubleshooting)
A: 안드로이드 Doze 모드(절전) 진입 시 백그라운드 위치 추적과 OBD 통신이 차단되는 문제였습니다. 이를 해결하기 위해 Foreground Service 타입을 'location'과 'connectedDevice'로 명시하고, WakeLock을 적절히 사용하여 OS에 의해 프로세스가 죽지 않도록 조치했습니다.

Q36. 팀 내 코드 리뷰 문화는 어땠나요?
A: GitHub Pull Request 템플릿을 만들어 '변경 목적', '테스트 방법'을 필수로 작성하게 했고, 1명 이상의 승인(Approve) 없이는 Merge 할 수 없도록 규칙을 정했습니다. 이를 통해 코드 품질을 유지하고 서로의 도메인 지식을 공유할 수 있었습니다.

Q37. 실제 테스트는 어떻게 했나요? (QA)
A: 팀원들의 차량(아반떼, 쏘나타, 스포티지) 3대를 섭외하여 실제 주행 테스트를 500km 이상 진행했습니다. 또한, ELM327 에뮬레이터를 활용해 '급가속', '고수온' 등 위험 상황을 가상으로 재현하여 엣지 케이스(Edge Case)를 검증했습니다.

Q38. 경쟁사 대비 기술적 해자(Moat)는 무엇인가요?
A: 단순 OBD 데이터 분석을 넘어, 시각(이미지)과 청각(소리)을 결합한 '멀티모달 AI 진단' 기술이 가장 큰 진입 장벽입니다. 또한, RAG를 통해 확보한 맞춤형 정비 지식 베이스는 시간이 지날수록 경쟁사가 모방하기 힘든 자산이 될 것입니다.

Q39. 서버 비용이 부담되지 않나요? (Cost Optimization)
A: 현재 AWS 프리티어와 RunPod(GPU 클라우드)의 저렴한 Spot 인스턴스를 활용해 월 5만 원 내외로 운영 중입니다. 스케일업 시에는 예약 인스턴스(RI)나 비용 효율적인 ARM 기반 프로세서(Graviton) 도입을 고려하고 있습니다.

Q40. 마지막으로 이 프로젝트를 한마디로 정의한다면?
A: "내 차안의 AI 주치의"입니다. 어렵고 낯선 자동차 정비를, 기술을 통해 가장 쉽고 친숙한 경험으로 바꾸는 서비스입니다.



================================================================================
9. 발표 핵심 포인트 (Q&A 대비)
================================================================================

Q1. 왜 이 프로젝트를 선택했나요?
A: 차량 정비는 전문가 의존도가 높고 비용 부담이 큽니다. AI로 사전 진단하면 사용자는 정비소 방문 전 문제를 파악하고, 정비소는 효율적인 정비 계획을 수립할 수 있습니다.

Q2. RAG를 왜 사용했나요?
A: 일반 LLM은 차량 정보를 모르거나 잘못된 정보를 생성합니다. RAG는 차량 매뉴얼을 검색 후 답변하므로 신뢰도가 높습니다.

Q3. 하이브리드 AI의 장점은?
A: 로컬 모델로 즉각 반응 + 클라우드 LLM으로 정밀 분석. 모바일 리소스 절약과 전문가 수준 진단을 동시에 달성합니다.

Q4. 확장성은?
A: MSA 지향 구조 (AI 서버 분리), RabbitMQ 비동기 처리, Kubernetes 수평 확장 가능

Q5. 비용 절감 전략?
A: Router 모델 필터링, Redis 캐싱, 향후 소형 모델 파인튜닝 계획

Q6. 보안은?
A: HTTPS(TLS) 암호화, AWS S3 Private Bucket, Pre-signed URL, JWT 토큰 인증

Q7. 모든 차종 지원?
A: 2008년 이후 모든 가솔린/디젤 차량 (OBD-II 표준). 전기차는 호환성 테스트 중
