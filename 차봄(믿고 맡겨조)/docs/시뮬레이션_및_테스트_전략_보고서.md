# OBD 에뮬레이터 고도화 및 테스트 가능성 분석 보고서

## 1. 프로젝트 전체 시뮬레이션 가능성 (Highly Possible)

현재 작성된 분석 문서를 바탕으로 에뮬레이터를 고도화하면 우리 프로젝트의 모든 주요 로직을 **PC만으로 100% 시뮬레이션**할 수 있습니다.

### 시뮬레이션 가능 항목:
- **신규 데이터 수집**: MAP, MAF, 흡기온도 등 신규 PID 응답을 에뮬레이터에 추가하여 앱이 이를 정상적으로 수집/저장하는지 확인 가능
- **DTC 및 MIL 알림**: 에뮬레이터에서 가상의 고장코드(예: P0115)를 응답하게 설정하고, 앱이 이를 감지하여 AI 서버로 보고하는 전체 플로우 테스트 가능
- **오프라인 큐 및 멱등성**: 네트워크 단절 시나리오를 시뮬레이션하여 데이터가 SQLite에 쌓이고, 재연결 시 `batchId`를 가지고 중복 없이 백엔드에 저장되는지 검증 가능
- **자동 주행 종료**: 에뮬레이터에서 RPM/Speed를 0으로 3분간 유지했을 때 앱이 스스로 세션을 종료하고 데이터를 서버로 전송하는지 테스트 가능

## 2. 다이소 BLE 5.0 동글 테스트 가능성 (Conditional)

다이소 동글을 이용한 테스트는 기술적으로 가능하지만, 몇 가지 허들이 있습니다.

### 도전 과제:
- **역할 문제**: 일반적인 PC용 블루투스 동글은 'Master(Central)' 역할입니다. 하지만 스마트폰이 PC를 OBD 스캐너로 인식하려면 PC가 **'Peripheral'** 역할을 해야 합니다.
- **프로토콜 구현**: 단순 시리얼 포트 연결이 아니라, BLE의 특성인 **GATT Service/Characteristic**을 ELM327 BLE 표준에 맞춰 PC에서 구현해야 합니다.

### 권장되는 하이브리드 테스트 전략:
1. **로직 검증 (Virtual Serial)**: `com0com` 등을 이용해 PC 내부에서 `에뮬레이터 <-> 테스트 앱` 간의 데이터 정합성을 먼저 완벽히 맞춥니다.
2. **연결성 검증 (SPP 모드 캐스팅)**: 만약 동글이 Classic BT(SPP)를 지원한다면 윈도우의 '고급 블루투스 설정'에서 수신/발신 COM 포트를 생성하여 실제 스마트폰과 무선으로 연동해볼 수 있습니다.
3. **BLE 최종 검증**: 실제 BLE 통신 환경을 완벽히 재현하려면, PC 에뮬레이터 대신 **ESP32**나 **실제 BLE OBD 고속 스캐너**를 사용하는 것이 훨씬 빠르고 정확한 결과를 얻을 수 있습니다.

## 3. 실제 차량 연동 테스트의 필요성

에뮬레이터는 '이상적인' 환경을 시뮬레이션하지만, 실제 차량 테스트는 다음 이유로 반드시 필요합니다.

### 실제 환경에서의 변수:
- **차량별 PID 지원 차이**: YF소나타와 제네시스가 지원하는 PID 목록과 응답 방식이 다를 수 있습니다. (우리가 만든 `disabledPids` 로직이 여기서 실제 효과를 봅니다.)
- **통신 노이즈 및 지연**: 블루투스 연결 환경에 따른 데이터 손실이나 지연(Latency)은 에뮬레이터에서 재현하기 어렵습니다.
- **프로토콜 호환성**: 대부분의 차량이 ISO 15765-4 (CAN)를 사용하지만, 방식(11bit/29bit, 250k/500k)에 따라 실제 동글이 연결을 잡는 시간이 다를 수 있습니다.

---

## 결론 및 제안

1. **에뮬레이터 고도화**: 시뮬레이션 가능성이 매우 높으므로, 고도화를 통해 **개발 효율성을 200% 이상** 끌어올릴 수 있습니다. (DTC 보고, 배치 처리 등 어려운 로직을 방 안에서 다 짤 수 있습니다.)
2. **다이소 동글**: "무선으로 연결된다"는 느낌을 확인하는 용도로는 클래식 BT로 시도해보시고, BLE 개발은 전용 장비를 권장합니다.
3. **전략**: `에뮬레이터로 로직 완성` -> `동글 테스트 (SPP)` -> `실차 테스트` 순으로 진행하시는 것을 제안드립니다.
