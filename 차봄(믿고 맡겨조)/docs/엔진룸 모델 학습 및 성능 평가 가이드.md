# YOLO 엔진룸 모델 학습 및 성능 평가 가이드 (상세설명 포함)

이 문서는 `ai/scripts/vision/train_engine.py`를 기반으로 엔진룸 부품 탐지 모델(YOLO)을 학습하고, 성능을 평가하는 방법을 설명합니다.
> **팁**: AI가 익숙하지 않은 분들을 위해, 어려운 용어 옆에 쉬운 설명을 덧붙였습니다. 

## 1. 개요
이 스크립트는 **엔진룸 내의 미세한 부품(Small Objects)**을 탐지하기 위해 최적화된 설정을 포함하고 있습니다.
- **기본 모델**: YOLO11m (Medium) + P2 Head (작은 물체 탐지 성능 강화)
- **학습 데이터**: `ai/data/yolo/engine/engine_merged.yaml`
- **이미지 크기**: 1280 (고해상도 학습)

---

## 2. 실험 설계: 베이스라인 vs 도전자 (Baseline Definition) 🥊

보여주신 이미지처럼, 우리도 명확한 **비교군(Baseline)**과 **도전자(Challenger)**를 설정하고 실험하는 것이 좋습니다.

| 역할 | 모델 정의 (추천) | 설명 | 목표 |
| :-- | :-- | :-- | :-- |
| **Baseline**<br>(기준점) | **YOLOv8n** (Nano) 또는<br>**Pretrained Model** | - 가장 가볍고 빠른 모델<br>- 또는 학습 전 순수 상태 | "최소한 이 점수는 넘어야 한다."<br>(기본 성능 확인) |
| **Challenger 1**<br>(주력) | **YOLO11m** (Medium) | - 우리가 현재 학습시키려는 모델<br>- 속도와 성능의 균형 | **Baseline 대비 압도적 성능 향상**<br>(실사용 목표) |
| **Challenger 2**<br>(고성능) | **YOLO11x** (X-Large)<br>**RT-DETR** | - 컴퓨터가 허락하는 한<br>가장 크고 똑똑한 모델 | **Challenger 1이 놓치는 미세 결함 잡기**<br>(성능 끝판왕) |

### 2.1 Baseline 성능 측정하기
학습을 시작하기 전에 "학습 안 한 상태"의 점수를 먼저 봅니다. 이게 우리의 바닥(Baseline)입니다.

```bash
# 베이스라인 점수 확인
python ai/scripts/vision/train_engine.py --mode baseline
```
- **실행 결과**: `mAP50` 점수가 나옵니다. (이 점수가 바로 깨야 할 기록입니다!)

## 3. 모델 학습 (Training)

Baseline을 확인했다면, 실제로 내 데이터를 공부시켜서 모델을 미세 조정(Fine-tuning)합니다.

### 3.1 학습 실행 명령어
```bash
# 기본 설정으로 학습 (Epochs: 200, Batch: 16)
python ai/scripts/vision/train_engine.py --mode train

# 설정을 바꿔서 실행하고 싶을 때
python ai/scripts/vision/train_engine.py --mode train --epochs 300 --batch 8
```

### 3.2 주요 설정값 (하이퍼파라미터) 설명
코드 내 `[Configuration]` 섹션에서 기본값을 변경할 수 있습니다.

| 변수명 | 쉬운 설명 | 추천 값 |
| :--- | :--- | :--- |
| `IMG_SIZE` | **이미지 크기**. 클수록 작은 부품을 잘 보지만, 속도가 느려집니다. | `1280` (권장 - 눈을 크게 뜨고 봄) |
| `BATCH_SIZE` | **한 번에 공부할 양**. 컴퓨터 메모리가 부족하면 줄여야 합니다. | `16` (12GB VRAM), `8` (6GB VRAM) |
| `EPOCHS` | **공부 횟수**. 문제집을 처음부터 끝까지 몇 번 볼 거냐는 뜻입니다. | `150` ~ `300`번 |
| `LR0` | **학습 속도**. 너무 빠르면 대충 배우고, 너무 느리면 속 터집니다. | `0.001` (기본값 추천) |

### 3.3 학습 결과물
학습이 끝나면 다음 위치에 파일이 생깁니다.
- **최종 두뇌 파일**: `ai/weights/engine/best_engine11m.pt` <- 이게 학습 결과물을 저장하는거예요 성빈님
- **성적표**: `ai/runs/engine_model/run/` (그래프와 표) <- 이건 문서로 정리한거구요 성빈님
- 학습 결과물들은 gitignore에 추가되어 있어요 git에 올라가지 않아요

## 6. 결함 탐지(Defect Detection) 모델 가이드

부품이 "있다/없다"를 찾는 것을 넘어, 그 부품이 **정상인지 불량인지(누유, 파손 등)** 판별하는 방법입니다.

사용자의 요청대로 다음과 같은 결과를 얻기 위한 방법입니다.
```json
{
  "defect_label": "Oil_Leakage",
  "defect_category": "LEAK"
}
```

### 6.1 추천 모델 리스트 <PatchCore를 썼던 이유는 정상 사진만 있어도 됨>

### 6.2 [옵션 1] 학습 방법 및 데이터 구성 (YOLO Classification - 추천)
컴퓨터에게 "이게 누유야", "이게 정상이야"라고 알려주려면 **폴더 정리**를 잘해야 합니다. 아래 그림처럼 폴더를 만들고 사진을 넣어주세요.


```
ai/data/anomaly/
├── Battery/              (부품 이름)
│   ├── train/            (공부용)
│   │   ├── good/         (정상 사진들 - 컴퓨터야 이게 정상 배터리야)
│   │   └── bad_leak/     (누액 사진들 - 이건 누액된 거야)
│   └── test/             (시험용)
│       ├── good/
│       └── bad_leak/
│
├── Engine_Oil_Cap/       (부품 이름)
│   ├── train/
│   │   ├── good/         (정상 사진들)

│   └── ...
```

> **꿀팁**: 폴더 이름(`bad_leak`, `bad_open`)이 나중에 AI가 뱉어내는 정답 이름이 됩니다. 결과를 **"Oil_Leakage"**로 받고 싶으면 폴더 이름을 **"Oil_Leakage"**로 만드세요!

### 6.3 학습 실행 방법 (Classification)

# epochs: 공부 횟수
model.train(data='ai/data/anomaly/Battery', epochs=100, imgsz=224)
```

### 6.4 [옵션 2] 학습 방법 및 데이터 구성 (YOLO Detection - 위치 추적)

만약 **"딱 여기(좌표)가 누유됐어!"**라고 박스를 쳐주고 싶다면 이 방법을 쓰세요. **라벨링(박스 그리기)** 작업이 필요합니다. -> 이건 좀 어려울 수 있어요 성빈님 ㅜ

#### 1) 데이터 준비 (라벨링 필수)
이미지와 함께 **정답 파일(.txt)**이 있어야 합니다.
- 라벨링 툴(LabelImg, Roboflow 등)을 써서 결함 부위에 박스를 그려야 합니다.
- 폴더 구조:
  ```
  ai/data/defect_det/
  ├── images/
  │   ├── train/ (이미지 파일들)
  │   └── val/
  └── labels/
      ├── train/ (박스 좌표가 적힌 .txt 파일들)
      └── val/
  ```

#### 2) 데이터 설정 파일 (data.yaml) 만들기
`ai/data/defect_det/data.yaml` 파일을 직접 만들어야 합니다.
```yaml
path: ai/data/defect_det
train: images/train
val: images/val
names:
  0: Oil_Leakage
  1: Rust_Corrosion
  2: Scratch
```

#### 3) 학습 명령어
`classify`가 아니라 **`detect`** 명령어를 씁니다.
```bash
# YOLOv8n (Detection 모델) 학습
yolo detect train data=ai/data/defect_det/data.yaml model=yolov8n.pt epochs=100 imgsz=640
```

## 7. 심화: 고성능 모델 추천 및 전략

### 7.1 고성능 모델 리스트 (더 똑똑한 뇌)

| :-- | :-- | :-- |
| **YOLO11x** | **전교 1등**. 제일 똑똑하지만 머리가 커서 느림. | RTX 3050에서는 좀 힘듦. (메모리 부족할 수 있음) |
| **YOLO11l** | **반장**. 1등(x)보단 못하지만 꽤 똑똑함. | 해볼 만함. (이미지 크기 줄이면 가능) |
| **YOLO11m** | **우등생**. 적당히 똑똑하고 빠름. | **지금 추천하는 모델!** (RTX 3050에 딱) |
| **YOLO11s** | **날쌘돌이**. 머리는 좀 나빠도 엄청 빠름. | 아주 잘 돌아감. (메모리 걱정 없음) |

### 7.2 이상 유형별 탐지 데이터 가이드 (Detailed Defect Guide) 🔎

사용자가 제공한 이상 유형 리스트를 바탕으로, **어떤 특징을 잡아야 하는지** 구체적으로 정리했습니다.

#### 1. 부품별 이상 유형 및 촬영 포인트 (Image Ideas)

| 부품 | 이상 유형 | 이미지 아이디어 (촬영 팁) | 탐지 난이도 |
| :-- | :-- | :-- | :-- |
| **ABS_Unit** | 균열, 깨짐 | 모서리 균열, 케이스 찌그러짐 | 🟢 쉬움 (형상 변화) |
| **Air_Filter_Cover** | 틈/열림 | 커버가 완전히 닫히지 않은 상태, 유격 발생 | 🟡 보통 (각도 중요) |
| **Battery** | 부식, 누액 | 단자 주변 하얀 가루(부식), 흘러내린 액체 자국 | 🟢 쉬움 (색상/질감) |
| **Brake_Fluid** | 누수, 변색 | 탱크 바닥 흘러내린 흔적, 갈색 변색(탁함) | 🟡 보통 (조명 중요) |
| **Engine_Oil_Fill_Cap** | 열림, 오염 | 뚜껑이 열려있거나(검은 구멍), 주변 기름 번짐 | 🟢 쉬움 (형상/색상) |
| **Radiator** | 누수, 변형 | 냉각수 얼룩, 라디에이터 핀이 휘어짐 | 🟢 쉬움 (패턴 변화) |
| **Engine_Cover** | 찌그러짐 | 엔진 커버 플라스틱 손상, 긁힘 자국 | 🟢 쉬움 (빛 반사) |
| **Windshield_Wiper_Fluid** | 누수, 깨짐 | 파란색 액체 누수, 탱크 균열, 캡 분리 | � 쉬움 (색상 명확) |

#### 2. 감지 특징별 분류 (Detection Strategy)

부품의 성격에 따라 AI가 "무엇을 보고 판단하는지"가 다릅니다. 이 특징을 살려서 사진을 찍어야 정확도가 올라갑니다.

*   **A. 외형 변화가 명확한 부품 (Shape-based)** 📐
    *   **대상**: `ABS_Unit`, `Radiator`, `Engine_Cover`, `Engine_Oil_Fill_Cap`, `Air_Filter_Cover`
    *   **특징**: 찌그러지거나 깨져서 **모양 자체가 변함**.
    *   **전략**: 그림자가 지면 모양이 잘 보입니다. 다양한 각도에서 "변형된 모습"을 많이 찍어주세요.

*   **B. 색상/질감 변화가 핵심인 부품 (Color/Texture-based)** 🎨
    *   **대상**: `Battery` (부식/누액), `Brake_Fluid` (변색), `Windshield_Wiper_Fluid` (색상)
    *   **특징**: 모양은 그대로인데 **색깔이 변하거나 얼룩이 생김**.
    *   **전략**: **조명(Light)**이 생명입니다! 너무 어둡거나 역광이면 색 구분이 안 됩니다. 밝은 곳에서 선명하게 찍어주세요.

#### 💡 데이터 수집 팁
> **Tip**: 이상 이미지가 많을수록 PatchCore(이상 탐지) 성능은 높지만, **10~20장 이상**만 있어도 학습이 가능합니다. 너무 부담 갖지 말고, 대표적인 불량 사진부터 모아보세요.

## 8. 심화: PatchCore 이상 탐지 모델 학습 가이드 (train_anomaly.py)

앞서 설명한 YOLO(Classification) 방식 외에, **"정상 사진만으로 학습"**하는 **Anomaly Detection (PatchCore)** 방식도 준비되어 있습니다.
이 방식은 **불량 데이터가 아직 하나도 없을 때** 유용합니다.

### 8.1 언제 쓰나요?
- **정상 사진은 많은데 불량 사진이 거의 없을 때.**
- "이게 불량이야"라고 알려주지 않아도, **"평소랑 다르면 다 불량이야"**라고 판단하게 하고 싶을 때.

### 8.2 준비하기 (폴더 만들기)
스크립트가 자동으로 폴더를 만들어줍니다.

```bash
# 폴더 구조 자동 생성
python ai/scripts/vision/train_anomaly.py --mode setup
```
- 실행하면 `ai/data/anomaly/` 아래에 부품별 폴더가 쫙 생깁니다.
- **[할 일]**: `ai/data/anomaly/[부품명]/train/good/` 폴더에 **정상 사진**만 넣어주세요. (불량 사진 넣으면 안 됩니다!)

### 8.3 학습 실행하기
사진을 넣었으면 이제 학습해봅시다.

**1) 특정 부품 하나만 학습하기 (예: 배터리)**
```bash
python ai/scripts/vision/train_anomaly.py --part Battery
```

**2) 전체 부품 한방에 학습하기**
```bash
python ai/scripts/vision/train_anomaly.py --all
```

### 8.4 만약 에러가 난다면? (Simple 모드)
`anomalib` 라이브러리 설치가 까다로울 수 있습니다. 에러가 나면 `--simple` 옵션을 붙여보세요. 복잡한 설치 없이 바로 돌아갑니다.

```bash
# 간편 모드로 배터리 학습
python ai/scripts/vision/train_anomaly.py --part Battery --simple
```

### 8.5 결과 확인
학습이 끝나면 다음 위치에 모델이 저장됩니다.
- 이 파일이 있으면 서버가 자동으로 "이상 탐지" 기능을 켭니다.

## 9. 데이터 부족(Data Scarcity) 해결 꿀팁 🍯

엔진룸 결함(누유, 부식 등) 사진은 구하기 정말 힘듭니다. 데이터가 부족할 때 쓸 수 있는 비기들을 전수합니다.

### 9.1 생성형 AI로 '가짜 불량' 만들기 🤖
요즘 AI(Stable Diffusion, DALL-E 3)는 "누유된 엔진" 사진을 기가 막히게 그려줍니다.
*   **프롬프트(명령어) 예시**:
    *   `"Car engine bay, oil leak stain on the valve cover, realistic, detailed"` (엔진룸 누유)
    *   `"Car battery terminals with white corrosion powder, close up"` (배터리 부식)
    *   `"Cracked plastic engine cover, dust and grime"` (커버 깨짐)
*   **활용**: 이렇게 만든 이미지를 학습 데이터에 섞어주면, AI가 "아, 이런 게 불량이구나" 하고 배웁니다.

### 9.2 Copy-Paste (오려 붙이기) 신공 ✂️
포토샵이나 그림판으로 "누유 자국"이나 "하얀 부식 가루"만 오려내세요.
*   **방법**:
    1.  누유 사진 1장에서 "기름 자국"만 오려냅니다. (PNG 투명 배경)
    2.  정상 엔진 사진 100장에다가 이 "기름 자국"을 여기저기 합성합니다.
    3.  순식간에 불량 데이터 100장이 생깁니다!

### 9.3 구글 & 유튜브 털기 🌍
직접 찍으러 다니기 힘들면 인터넷을 활용하세요.
*   **검색어**: `Engine oil leak repair`, `Radiator crack`, `Battery corrosion cleaning`
*   **유튜브**: 정비소 아저씨들이 고치는 영상을 캡처하세요. 실제 현장 데이터라 아주 좋습니다.

## 10. AI 학습용 데이터(라벨링) 가이드 🏷️

데이터를 모았으면, 이제 컴퓨터가 알아먹게 **"이게 배터리야", "이게 누유야"** 하고 표시를 해줘야 합니다.

### 10.1 어떤 라벨링이 필요한가요?

| 모델 종류 | 필요한 라벨링 방식 | 설명 | 난이도 |
| :-- | :-- | :-- | :-- |
| **Classification**<br>(분류) | **폴더 정리** 📁 | 그냥 폴더(`bad_leak`)에 사진 넣으면 끝. | ⭐ (아주 쉬움) |
| **Detection**<br>(위치 추적) | **박스 그리기** 📦 | 사진 속 부품에 네모 박스를 그려야 함. | ⭐⭐⭐ (손이 많이 감) |

### 10.2 박스 그리기 도구 (Detection용)

만약 **"딱 여기(좌표)가 불량이야!"**라고 찾고 싶다면(`6.4 섹션`), 아래 도구를 쓰세요.

#### 1) Roboflow (추천! 👍)
웹사이트에서 마우스로 슥슥 그리면 됩니다.
- **주소**: [https://roboflow.com/](https://roboflow.com/)
- **장점**: 설치 불필요. 팀원들과 같이 할 수 있음. 형식을 자동으로 변환해줌(**YOLO v8**로 선택).

#### 2) LabelImg
컴퓨터에 설치해서 쓰는 고전적인 프로그램입니다.
- **설치**: `pip install labelimg`
- **주의**: 저장할 때 꼭 **YOLO 포맷**(.txt)으로 저장하세요. (XML은 안 됩니다!)

### 10.3 라벨링 잘하는 법 (Best Practices)
1.  **여백 없이 꽉 채워서!**: 부품이나 결함 부위에 딱 맞게 박스를 치세요. 헐렁하면 AI가 헷갈려합니다.
2.  **가려진 것도 상상해서**: 엔진 호스에 가려진 부품이라도, 보이는 만큼 최대한 박스를 쳐주세요.
3.  **일관성**: "이 정도 얼룩도 누유로 칠까?" -> 팀원들과 기준을 통일하세요. 안 그러면 AI가 혼란스러워합니다.

## 11. 심화: 전처리 및 증강 전략 (Preprocessing Strategy) 🎨

엔진룸은 외관과 달리 **색상(Color)**이 매우 중요합니다. 함부로 색을 바꾸면 안 됩니다.

### 11.1 엔진룸(Engine) 핵심 전략
기름 누유(오일색), 냉각수(분홍/초록), 배터리 부식(흰색) 등 **색깔**이 곧 정답입니다.

| 설정 항목 | 추천 값 | 이유 |
| :-- | :-- | :-- |
| **색상 (HSV-Hue)** | **거의 끄기** (0.01) | **절대 주의!** 냉각수 색깔이 바뀌면 안 됩니다. 색상은 원본 그대로 두세요. |
| **밝기 (HSV-Val)** | **밝게** (+0.4) | 엔진룸 깊숙한 곳은 어둡습니다. 밝기를 올려서 어둠 속 부품을 찾아내세요. |
| **회전 (Rotate)** | **자유롭게** (±20도) | 엔진룸 사진은 위에서, 옆에서 다양하게 찍습니다. 자유로운 회전이 도움됩니다. |
| **노이즈 (Noise)** | **추가** | 먼지나 기름때가 많은 환경입니다. 노이즈를 섞어서 더 강하게 학습시킵니다. |

### 11.2 코드 적용 예시
`train_engine.py`에서 `model.train()` 함수 사용 시:

```python
model.train(
    data=data_yaml,
    epochs=150,
    imgsz=1280,       # 1. 해상도: 작은 부품을 찾아야 하므로 1280 권장! (메모리 부족하면 640)
    hsv_h=0.01,       # 🚨 색상 변화 최소화 (냉각수/오일 색깔 보존)
    hsv_s=0.5,        # 채도 변화 적당히
    hsv_v=0.4,        # 밝기 변화 (어두운 곳 대비)
    degrees=15.0,     # 각도 비틀기 허용
    scale=0.5,        # 크기 변화 허용 (줌인/줌아웃)
    mosaic=0.5        # 모자이크는 적당히 (너무 복잡하면 방해됨)
)
```

## 12. 심화: 이미지 크기(Resolution) 추천 가이드 📏

"무조건 1280이 좋은가요?" -> **아닙니다!** 상황에 따라 정답이 다릅니다.

| 상황 (Scenario) | 추천 크기 | 이유 |
| :-- | :-- | :-- |
| **1. 전체 사진에서 부품 찾기**<br>(Detection) | **1280 px** | 사진은 큰데 부품(오일캡)은 손톱만합니다. 해상도가 낮으면 부품이 점으로 보여서 못 찾습니다. **크게 봐야 보입니다.** |
| **2. 이미 잘라낸 부품 사진**<br>(Classification / Anomaly) | **224 ~ 640 px** | 이미 배터리만 딱 잘라놓은 사진은 작습니다(300px 등). 이걸 억지로 1280으로 늘리면 **깍두기 현상(화질 저하)**만 생겨서 오히려 성능이 떨어집니다. **원본 크기와 비슷하게 쓰세요.** |

> - `train_anomaly.py` (잘린 부품 분류): **224** 또는 **원본 크기** 추천

## 13. [부록] 모델 출력 데이터 예시 (JSON Structure)

서버에서 클라이언트로 전달되는 최종 JSON 데이터 형식입니다.

### 13.1 엔진룸 진단 (SCENE_ENGINE)

```jsonc
{
  "status": "WARNING", // 종합결론 : NORMAL, WARNING, CRITICAL, ERROR(판단불가 -> LLM)
  "analysis_type": "SCENE_ENGINE",
  "category": "ENGINE_ROOM",
  "data": {
    "vehicle_type": "ICE", // ICE(내연기관) / EV(전기차) - 감지된 부품 기반 분류
    "parts_detected": 2, // 감지된 부품 수
    "anomalies_found": 1, // 이상 징후 부품 수
    "results": [
      {
        "part_name": "Engine_Oil_Fill_Cap",
        "bbox": [697, 227, 875, 348], // [x1, y1, x2, y2]
        "confidence": 0.92, // 탐지 신뢰도
        "is_anomaly": true, // 이상 징후 여부
        "anomaly_score": 0.85, // 이상 점수
        "threshold": 0.7, // 임계값
        "defect_label": "Oil_Leakage", // 결함 명칭
        "defect_category": "LEAK", // LEAK, CORROSION, PHYSICAL, WEAR, UNKNOWN
        "severity": "WARNING" // 개별 부품 심각도
      }
    ]
  }
}
```

#### 💡 심각도 기준 (Severity)
| 레벨 | 점수 | 설명 |
| :-- | :-- | :-- |
| **CRITICAL** | 9점 | 엔진 과열, 오일 부족 등 **즉시 정차**가 필요한 치명적 결함 |
| **WARNING** | 5점 | 냉각수 부족, 미세 누유 등 **빠른 시일 내 점검** 권장 |
| **NORMAL** | 0점 | 이상 징후 없음, 안전 운행 가능 |

#### 📋 부품 분류 기준 (Vehicle Type Reference)
*   **ICE (내연기관) 전용**:
    *   `Engine_Oil_Fill_Cap`, `Engine_Oil_Dip_Stick`, `Engine_Cover`, `Oil_Filter`
    *   `Alternator`, `Transmission_Oil_Dip_Stick`, `ATF_Oil_Reservoir`, `Clutch_Fluid_Reservoir`, `Cold_Air_Intake`
*   **EV (전기차) 전용**:
    *   `Electric_Motor`, `Inverter_Coolant_Reservoir`, `Secondary_Coolant_Reservoir`, `Intercooler_Coolant_Reservoir`
*   **공통 부품**:
    *   `Battery`, `Radiator`, `Radiator_Cap`, `Engine_Coolant_Reservoir`, `Brake_Fluid`
    *   `Windshield_Wiper_Fluid`, `Fuse_Box`, `ABS_Unit`, `Power_Steering_Reservoir`
