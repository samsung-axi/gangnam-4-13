# AI 진단 시스템 비동기/논블로킹 전환 로드맵

이 문서는 현재의 블로킹(Blocking) 방식 진단 시스템을 대규모 사용자를 수용할 수 있는 논블로킹(Non-blocking) 이벤트 기반 시스템으로 개선하기 위한 단계별 계획을 담고 있습니다.

---

## 🚀 5단계 로드맵

### 1단계: 번호표 시스템 도입 (식별자 Echo 구조) - [현재 진행 중]
- **목표**: 모든 요청에 고유 번호표(`sessionId`, `vehicleId`)를 붙여 전송.
- **핵심**: AI 서버가 결과를 돌려줄 때 이 번호표를 그대로 보게 하여, 백엔드가 누구의 데이터인지 즉시 알 수 있게 함.
- **전송되는 4개 식별자**:
    1. `vehicle_id`: 차량 고유 식별자 (모든 요청)
    2. `session_id`: 진단/분석 세션 고유 식별자 (모든 요청)
    3. `chunk_index`: 전체 데이터 중 현재 몇 번째 조각인지 (이상징후 분석 시)
    4. `total_chunks`: 전체 데이터 조각 개수 (이상징후 분석 시)
- **이점**: 로그 추적이 쉬워지고, 데이터 뒤섞임 사고를 완벽히 방지함.

### 2단계: 주소 알려주기 (Callback URL)
- **목표**: 요청 시 "다 되면 이 주소(API)로 전화해줘"라고 콜백 주소를 전달.
- **핵심**: AI 서버가 "접수 완료"만 즉시 응답하고 연결을 끊음.
- **이점**: 백엔드 일꾼(Thread)이 AI 서버를 기다리며 멈춰있지 않고 바로 자유로워짐.

### 3단계: 전용 수신 창구 개설 (Webhook Handler)
- **목표**: AI 서버가 비동기로 보내오는 결과를 수신할 전용 API 엔드포인트 구축.
- **핵심**: 수신된 `sessionId`를 바탕으로 분석 결과를 DB에 정확히 업데이트.
- **이점**: 백엔드 일꾼(Thread)이 AI 서버를 기다리며 멈춰있지 않고 바로 자유로워짐.

### 4단계: 퍼즐 완성기 구현 (Result Aggregation)
- **목표**: 시각/청각/이상징후 조각이 모두 모였는지 체크하는 로직 도입.
- **핵심**: 마지막 조각이 도착하는 순간 최종 종합 진단(Phase 2)을 자동으로 실행.
- **이점**: 파이프라인의 전 과정이 사람의 개입 없이 자동으로 흐름.

### 5단계: 실시간 상태 알림 (WebSocket/SSE)
- **목표**: 사용자가 새로고침하지 않아도 화면이 즉시 바뀌게 실시간 통신 적용.
- **핵심**: 백엔드에서 결과가 완성되는 즉시 프론트엔드로 신호를 쏨.
- **이점**: 사용자 경험(UX)이 극대화됨.

---

## 🛠️ 개발 환경 및 주의사항

- **로컬 테스트 시**: 2단계(Callback) 이후부터는 AI 서버가 내 컴퓨터(localhost)로 전화를 걸 수 있어야 합니다. 
    - 만약 AI 서버가 외부에 있다면 **ngrok** 같은 도구를 사용하여 내 로컬 주소를 외부로 노출시켜야 합니다.
    - AI 서버도 내 로컬에 같이 있다면 내부 IP나 localhost 주소로도 테스트가 가능합니다.
- **배포 환경**: 배포 후에는 고정된 도메인 주소를 콜백 주소로 사용하게 됩니다.
